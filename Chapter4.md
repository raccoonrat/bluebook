Chapter 4. Runtime Security Kernel: Multi-Layered Detection, Routing, and Explainable Execution

在 Unified AI Security Control Plane 的整体结构中，运行时安全并不是一个附属执行环节，更不是在模型输出之后附加的一层简单过滤。当前设计已经明确指出，系统真正要回答的问题不是“是否有一个 guard model”，而是：**运行时安全到底是如何一步步发生的，系统如何从输入、语言识别、预处理、轻量检测、智能路由、重模型升级，到最终决策执行形成一条工业级、可解释、可治理的防护链路。** 这一定义本身已经表明，Runtime Security Kernel 在本体系中不应被理解为若干离散 detector 的堆叠，而应被理解为统一控制平面中的核心执行内核。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

当前设计对这一内核的结构给出了明确描述：运行时流程从输入进入系统开始，首先经过 `detect_languages()` 所代表的语言识别与语言门控；随后进入 **Normalization、Sanitization、Heuristics** 等预处理阶段；接着进入由大量 detector card 组成的 **lite detector shelf**；在中间由 **router / intelligent gate** 决定是走 fast path 直接输出，还是升级到 **heavy detector**；最后进入 **decision & action** 层，输出 **Scoring、Aggregation、Allow / Block / Redact / Degrade** 等结果。材料特别强调，这一视觉与逻辑重点不是“模块多”，而是要体现严格的工业流水线感，这意味着系统的核心并非单点判定，而是一个多阶段、可路由、可升级、可执行的连续决策过程。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这种结构首先说明，运行时安全的本质并不是“单模型替代一切”，而是**分层检测**。当前设计已经明确指出，系统采用的是 **rule-based、heuristics-based、proximity-based、regex-based、ML-based** 的混合架构，而不是单一 detector。由此，Runtime Security Kernel 的第一原则便是异构性：不同类型的检测机制在系统中承担不同角色，有些负责低成本、快速、显式的模式识别，有些负责更高成本、更高能力的语义判断，而整个运行时内核的任务，不是选择某一种机制替代其他机制，而是把这些异构组件组织为统一的检测与执行序列。也正因为如此，本章讨论的对象并不是某个 detector family 本身，而是异构检测机制如何在统一内核中被调度、被组合并最终进入一致的决策层。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

在这一分层结构之中，**language gate** 与预处理阶段具有基础性地位。当前设计明确将 `detect_languages()` 放在最前端，并在视觉上把第一关定义为语言门控：支持的语言进入对应 bundle，不支持的语言被阻断；紧接其后的是 **Normalization / Sanitization / Heuristics**。这一结构说明，运行时防护并不是从“内容危险性判断”开始，而是从“输入是否能够被系统正确识别、规范化并进入可处理空间”开始。也就是说，运行时安全的第一步不是高层语义判定，而是将原始输入转化为可进入后续检测链路的标准化对象。尤其在当前系统同时强调多语言支持与中国/多语安全能力的情况下，这一前置阶段并非简单文本处理，而是整个内核能否正确运行的前提条件。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

在预处理之后，系统进入 **lite detector shelf**。当前设计特别强调，这一层由“大量 detector card”构成，是运行时内核中的轻量检测层。从体系定位上看，这一层并不是低级或次要能力，而是决定系统吞吐、时延与大多数常规判断路径的主要执行面。材料已经说明，小 detector 可以承担多数情形下的判断，而当其“不确定”时，再升级到 SLM / heavy detectors。由此，lite detector 层的作用不是“做简单检查”，而是作为整个运行时内核的第一道广覆盖决策层：在控制成本与时延的同时，尽可能完成高频、常见、结构明确的安全判断，并为后续的升级路径提供必要的初始信号。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

正因为轻量层并不负责处理所有情况，**intelligent gate / router** 才成为整个内核中最关键的中间机制之一。当前设计已经明确指出，系统支持根据 **输入语言、需求和置信度** 决定何时使用 heavier detectors；当小 detector 不确定时，再升级到 SLM / heavy detectors。与此同时，运行时架构图把中间节点明确命名为 **router / intelligent gate**，并将其定义为决定 fast path 还是升级路径的控制中枢。这说明，内核中的路由并不是一个简单的 if-else 逻辑，而是系统在运行时对检测资源、语义复杂度与风险不确定性进行组织的核心位置。也正是在这里，运行时安全开始表现出“决策系统”的特征：并非所有输入都接受同等深度的判断，而是依据当前可得信号进行层次化分流。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这种由 lite 到 heavy 的升级路径，使得 **heavy detector block** 在系统中具有明确而克制的定位。当前设计并未把 heavy models 描述为默认必经的主路径，而是把它们定义为高成本、高能力的升级路线，用于承接轻量层无法稳定判定或需要更高表达能力的情形。这一设计直接体现了运行时内核的工程理性：系统既不假定重模型是唯一可信路径，也不假定轻量组件可以覆盖全部安全情形，而是通过中间路由机制在两者之间建立受条件触发的升级关系。对于企业级系统而言，这种结构的重要性在于，它允许安全能力、资源成本与时延开销在同一内核中实现组织化平衡，而不是通过静态配置将所有请求压入同一成本层级。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

当多阶段检测与路由完成后，系统进入 **Decision Layer**。当前设计在这一部分给出的表述非常明确：最终输出不是单纯的 allow / block，而是 **Scoring、Aggregation、Allow / Block / Redact / Degrade** 的综合决策；同时，材料还明确指出，xCloud 场景要求 **explainable blocking reasons**，而现有系统已强调 **scoring system** 与 **explainability engine**。这表明，Decision Layer 的职责并不只是“给出结论”，而是把前面各阶段的检测结果、聚合逻辑与动作策略收敛为一个可解释、可执行、可进入后续 evidence chain 的运行时决定。换言之，运行时安全在这里不再只是拦截，而是开始产生控制平面意义上的标准化判断结果。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这一定义还进一步说明，运行时动作本身必须被视为控制策略的一部分，而不是检测后的附带处理。当前设计已经把最终动作明确列为 **Allow / Block / Redact / Degrade**，并将其放在 decision & action 层的最右端，这意味着系统并不把所有风险都压缩为“通过 / 不通过”的二元判断，而是承认在企业级部署中，不同风险情形可能需要不同的控制方式。有的输入可以被允许，有的需要被阻断，有的需要被删改，有的则需要在能力层面进行降级处理。由此，Runtime Security Kernel 并不只是产生安全标签，而是把安全判断直接转化为可执行的控制动作，这也使其成为后续风险自适应控制选择与证据治理的天然上游。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

当前设计还明确指出，Input Controls 的整体定位是 **模块化、多语言、可解释、带统一 scoring** 的能力，并且与 Evaluation Framework 集成。这个表述对于理解运行时内核尤为关键：它说明 Runtime Security Kernel 并不是脱离控制平面独立存在的“执行插件”，而是一个已经被统一 scoring、统一分类与统一评测框架约束的系统构件。模块化意味着 detector 与执行组件可以被系统性接入；多语言意味着 language gate 与 detector bundles 必须内生支持不同语言环境；可解释意味着 decision layer 的输出不是黑箱阻断；统一 scoring 则意味着运行时判断能够与上游 benchmark governance、下游 evidence 与 release gating 保持一致的度量语言。这样一来，运行时内核便不只是执行层，而是控制平面统一语义—判断—证据链条中的中间核心。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

从理论角度看，当前设计已经非常明确地指出，运行时安全最适合被升格为 **Composable Runtime Defense** 或 **Multi-stage Safety Cascade** 的理论问题。材料中具体列出的研究方向包括：多阶段 detector cascade 的误差传播如何分析；在 **latency / cost / false positive** 约束下，light→heavy 的升级策略如何最优；以及 **sanitization、pattern detector、ML detector、guard model** 这些异构组件如何组合成可分析的防线。这里最重要的变化在于，运行时架构不再被理解为“工程技巧”，而是被正式视为一个可以研究其组合规律、误差行为与决策边界的科学对象。也就是说，Chapter 4 的理论价值并不在于把 pipeline 画得更复杂，而在于把企业级运行时防护提升为可分析、可优化、可解释的组合防御问题。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这也意味着，本章必须明确系统**不应如何被理解**。首先，它不应被理解为“一个大 guard model + 若干前处理”的方案，因为当前设计已经清楚地把 rule-based、heuristics-based、regex-based、ML-based 等多种机制并列为体系组成部分，并明确存在轻量层、路由层与重模型升级层；其次，它不应被理解为“单次判断管线”，因为现有架构中的 scoring、aggregation、explainability 与 action outputs 已经表明，系统关注的是可执行、可引用、可进入 evidence chain 的标准化结果；最后，它也不应被理解为局部过滤器，因为 Input Controls 已被定义为与 Evaluation Framework 集成的模块化能力，并位于统一 AI Security Control Plane 的中心判断层之中。只有在这些误解被排除之后，Runtime Security Kernel 才能被正确理解为企业级 AI 安全系统中的执行中枢，而不是一组局部拦截组件。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

综上所述，本章的核心结论是：**Runtime Security Kernel 是 Unified AI Security Control Plane 的运行时执行核心，它通过 language gate、Normalization、Sanitization、lite detectors、intelligent routing、heavy detector escalation 以及 decision & action layer，把输入处理、分层检测、升级判断、解释输出与控制动作组织为统一的工业级安全流水线。** 当前设计已经明确给出了这一内核的结构组成、判断逻辑与理论升格方向，包括模块化、多语言、统一 scoring、explainability、rule-based / heuristics-based / regex-based / ML-based 混合架构、基于语言 / 需求 / 置信度的智能路由，以及以 **Composable Runtime Defense / Multi-stage Safety Cascade** 为代表的研究命题。由此，Chapter 4 并不是对若干 detector 的说明，而是在蓝皮书中正式确立：企业级 AI 安全的运行时层必须被设计成一个可组合、可解释、可升级、可进入统一证据体系的安全执行内核。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)
