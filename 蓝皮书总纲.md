一、蓝皮书的总目标：从“能力集合”升级为“持续强化的 AI 安全科研—工程体系” 

1.1 总定位 

这份蓝皮书的目标，不应再表述为“做若干 detector、一个 evaluation 平台、以及若干中国区能力”，而应表述为： 

构建一个统一的 AI Security Control Plane，使风险测量、运行时控制、geo-specific intelligence、证据治理与上线放行形成一个闭环系统。 这一点与你材料中对系统边界的定义是一致的：中心是 Security Control Plane，上方是 Evaluation & Benchmark Governance，右侧是 xCloud / Execution Environment，右下是 Evidence & Release Gating，左下是 Chinese & Multilingual Safety Intelligence。 

1.2 蓝皮书应回答的核心问题 

这份蓝皮书要回答的，不是“DTL 有哪些能力”，而是： 

我们到底在构建什么样的 AI 安全系统；其核心理论对象是什么；它如何持续强化；以及它如何从 benchmark、判断、runtime API 到 evidence chain 形成统一能力。 这些问题在材料中已经被直接提出，并被作为系统化能力的关键技术方向。 

1.3 最终形态 

蓝皮书最终要把整个研发 program 定义成一个可持续强化的科学—系统母体： 

* 以 统一语义层 支撑安全/隐私/内容安全/泄漏/攻击等风险分类与策略表达。 

* 以 双轨评测底座 支撑全球安全与中国合规的并行评测与统一决策。 

* 以 运行时安全内核 支撑分层检测、智能路由与解释性动作执行。 

* 以 风险驱动闭环 支撑从 measurement 到 control selection，再到 evidence feedback 的持续强化。 

* 以 统一研发框架 保证系统突破、中文/合规突破、评测突破与理论突破在一个 program 中协同推进。 

二、蓝皮书建议采用的“六大研发项目”主结构 

下面这六个项目，不是彼此独立的模块，而是围绕同一个控制平面展开的六个主科研—研发方向。其中每个方向都对应你材料中已经明确存在的系统要素；我这里做的是项目化细化与技术蓝皮书化组织。 

项目 A：安全语义与治理底座（Security Semantics & Governance Foundation） 

A.1 项目目标 

建议定义：建立一套统一的安全语义与治理底座，把安全、隐私、内容风险、攻击、合规要求、评分体系、放行条件统一到一个可编排的治理对象中。 

这个项目直接对应材料中明确提出的三件事： 

* Input Controls 具备 standardized detection categories and scoring； 

* Evaluation 不能脱离 benchmark 与 acceptance contract； 

* Chinese & Multilingual Safety Intelligence 需要被纳入统一控制平面，而不是外围适配。 

A.2 核心基础理论 

建议主打理论主线： 

1. Compliance-aligned safety semantics：如何把不同监管、不同业务、不同模型场景的安全要求统一映射到一套安全语义。其概念基础已经在材料中被明确列为理论突破点。 

2. Geo-aligned safety semantics：如何把中国语义与全球安全稳健性放进同一评测范式与控制平面下比较。材料在双轨评测部分已直接提出这一升格方向。 

3. Compliance-to-Evaluation Compilation：如何把监管/合规要求编译成 test suite、aggregation logic 与 report template。该方向已在材料中直接作为研究建议出现。 

A.3 强化关键技术 

建议交付的关键技术： 

* Unified Safety Taxonomy（统一风险分类） 

* Policy Schema / Acceptance Contract Schema（策略与放行合同表达） 

* Geo-specific semantics 扩展（面向 TC260 / 中文 / 多语表达） 

这些技术方向与材料中“统一语义层”“benchmark governance”“acceptance contract”“geo-specific intelligence” 的表述一致。 

A.4 研发交付物 

建议的可交付成果： 

* 蓝皮书第一章：统一安全语义模型 

* 安全分类与评分规范 

* Acceptance Contract 规范 

* 中国区 / 全球双轨语义映射规范 

这些是基于材料中“统一 taxonomy、统一 scoring、双轨评测、统一决策与证据输出”的项目化落地。 

项目 B：双轨评测与决策底座（Dual-Track Evaluation & Decision Backbone） 

B.1 项目目标 

建议定义：建设一套共享执行底座的双轨评测系统，使 Global Safety & Robustness Track 与 China Regulatory Compliance Track（TC260） 在同一平台内运行、共享 run abstraction 与结果回溯能力，但保留不同的 test suite、aggregation logic 和 decision logic。 

这正是材料中对双轨评测的明确定义。 

B.2 核心基础理论 

建议主打理论主线： 

1. Multi-track evaluation under shared execution backbone：如何在共享执行引擎下保持不同监管目标的独立判断。材料明确指出双轨“共享底座、不同决策逻辑”。 

2. Release admissibility under heterogeneous risk semantics：如何把多轨评测结果统一进入 combined decision board 与 release decision。材料已明确提出“合流到 unified decision / evidence output”。 

B.3 强化关键技术 

建议交付的关键技术： 

* Shared evaluation backbone（Run / Target / Test Profile / Result Store / API / CI-CD） 

* ROW / TC260 双轨执行器 

* Unified decision board 

* 报告与回溯模板 

这些能力在材料中已经被逐项列出。 

B.4 研发交付物 

建议的可交付成果： 

* 双轨评测平台蓝皮书章节 

* Benchmark Constitution（评测宪法） 

* 每类模型 / 场景 / 地域的 Acceptance Contract 

* 回归测试与发布决策流程说明 

这些项目化建议直接服务于材料中“benchmark governance、release decision、audit” 的方向。 

项目 C：运行时安全内核（Runtime Security Kernel） 

C.1 项目目标 

建议定义：把当前的 pipeline 系统化、内核化，使运行时防护从“若干 detector”升级为一个可解释、可升级、可组合的安全内核。 

材料对这条 pipeline 的定义非常清晰：从输入进入 language gate，再经过 Normalization / Sanitization / Heuristics，进入 lite detector shelf，由 intelligent gate 决定 fast path 或升级到 heavy detector，最后进入 decision & action，输出 allow / block / redact / degrade。 

C.2 核心基础理论 

建议主打理论主线： 

1. Composable Runtime Defense：材料已直接建议把运行时安全升格为 “Composable Runtime Defense” 理论问题。 

2. Multi-stage Safety Cascade：材料已明确提出多阶段 detector cascade 的误差传播、升级策略最优与异构组件组合问题。 

3. Heterogeneous detector composition：材料明确说明当前系统是 rule-based、heuristics-based、proximity-based、regex-based、ML-based 的混合架构。 

C.3 强化关键技术 

建议交付的关键技术： 

* Kernel API / Policy Engine 

* Detector Registry（lite / heavy / multilingual / rule-based） 

* Aggregation + Scoring Engine 

* Explainable Decision Layer 

这些关键技术都与材料中“统一判断层”“统一 scoring”“explainable blocking reasons”一致。 

C.4 研发交付物 

建议的可交付成果： 

* Runtime Security Kernel 规范 

* Detector 接入标准 

* 路由与升级策略 

* 动作与解释输出格式 

这些交付物直接落在材料已定义的运行时流水线与解释性要求上。 

项目 D：风险自适应控制选择（Risk-Adaptive Control Selection） 

D.1 项目目标 

建议定义：把 Evaluation 从“离线测分”提升为“控制决策信号源”，并构建一套面向 runtime 的风险自适应控制选择机制。 

材料在“Risk Adaptation”一页中已经把这个闭环定义得很完整： 

* Measure：Benchmarking / compare variants； 

* Profile：转成 per-model / per-scenario risk profile； 

* Select：从 rule-based / lightweight / heavy 中选择控制级别； 

* Enforce：在 runtime 执行 allow / block / redact / degrade； 

* 右侧再接 Evidence & Release Gating，并反向反馈到 Measure。 

D.2 核心基础理论 

建议主打理论主线： 

1. Risk-Adaptive Control Selection：材料已直接指出这是最强的理论突破点。 

2. Selective classification / abstention：材料明确提到可上升为 selective classification / abstention。 

3. Constrained optimization under SLA / cost / risk：材料直接列出该方向。 

4. Multi-stage decision under uncertainty：材料同样将其列为该页的理论表达。 

D.3 强化关键技术 

建议交付的关键技术： 

* Risk Profile Schema 

* Control Selection Policy Engine 

* Base model capability / robustness / compliance risk / cost / latency / risk tolerance 联合建模 

* Runtime control ladder（rule / lite / heavy / degrade） 

这些技术要素都被材料明确列为控制选择的输入因子。 

D.4 研发交付物 

建议的可交付成果： 

* 风险画像模型 

* 场景化控制级别选择器 

* 模型—场景—控制的联合决策规范 

* 评测驱动控制策略的蓝皮书章节 

这些都直接来自材料中“evaluation-driven control selection”的项目化延展。 

项目 E：证据图谱与上线放行（Evidence Graph & Release Gating） 

E.1 项目目标 

建议定义：建设一个面向发布治理的证据图谱系统，使 benchmark、runtime judgment、audit artifact、release gating 构成统一的 machine-checkable evidence chain。 

材料已经明确强调：技术重点不是单 detector，而是**“从 benchmark → judgment → runtime API → evidence chain” 的统一系统化能力**。 

E.2 核心基础理论 

建议主打理论主线： 

1. Machine-checkable safety evidence：依据材料中的 evidence chain、release gating、audit artifact，建议将其升格为“可机读可校验的安全证据体系”。其概念基础来自材料中已明确的统一证据层与放行决策。 

2. Evidence-centric release governance：材料已把 result store、release decision、audit 输出整合为统一决策面板的一部分。 

E.3 强化关键技术 

建议交付的关键技术： 

* Evidence Schema 

* Release Gate Artifact

这些都与材料中“统一证据层”“release gating 与 audit artifact”的逻辑一致。 

E.4 研发交付物

三、把六个项目连接成一个“可持续强化”的科研—工程闭环 

蓝皮书最重要的一点，不是把六个项目并列摆开，而是明确它们之间的强化机制。 

材料已经给出非常清晰的闭环基础：Measure → Profile → Select → Enforce → Evidence & Release Gating → Feedback to Measure。 

因此，这份蓝皮书建议明确把整个 program 的持续强化机制写成： 

1. 语义与治理底座 定义安全对象、分类和放行契约。该层对应统一 taxonomy、benchmark governance 与 acceptance contract。 

2. 双轨评测底座 把全球安全与中国合规编译成共享底座上的可运行实验。该层对应 dual-track evaluation、shared backbone 与 unified decision。 

3. 运行时安全内核 将策略表达落实到 language gate、preprocess、lite/heavy detectors、decision action 等执行元件。 

4. 风险自适应控制选择 根据 risk profile、模型能力、成本、时延与风险容忍度进行控制级别选择。 

5. 证据图谱与发布治理 把实验结论与运行时动作转化为可回溯的 release gating 和 audit artifact。 

6. 中国与多语智能 持续把 geo-specific semantics 与真实本地风险现象反馈进 benchmark、runtime 与 release 决策。 

这六步共同构成一个不是“一次性交付”的系统，而是一个持续强化、持续校准、持续审计、持续上线治理的 AI 安全科研平台。其逻辑基础完全来自你材料中对统一控制平面与风险驱动闭环的定义。
