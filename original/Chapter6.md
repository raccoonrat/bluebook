Chapter 6. Evidence Graph, Release Gating, and Auditability

在 Unified AI Security Control Plane 的整体架构中，证据与放行并不是运行时动作之后的附属流程，也不是面向管理与审计的后置材料整理。当前设计已经明确指出，**Evidence & Release Gating** 是控制平面的核心组成部分之一，其职责是把判断结果转化为可追踪的证据与放行决策。由此，本章讨论的对象，不是一般意义上的日志系统或报表系统，而是整个控制平面如何把 benchmark、judgment、runtime enforcement 与 release decision 连接成统一的证据链。也正因为如此，当前设计特别强调，真正需要突出的关键技术并不是某个 detector，而是 **从 benchmark → judgment → runtime API → evidence chain 的统一系统化能力**。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这一表述首先改变了我们对“证据”的理解。在传统工程语境中，证据常常被理解为运行记录、审计日志或离线评测报告，它们更多扮演“发生过什么”的事后记录角色。然而在当前设计中，Evidence 并不是后验产物，而是控制平面中的制度性对象：它与 benchmark governance、runtime judgment、release gating 和 audit artifact 位于同一逻辑链条之中，并被设计为统一决策与持续改进机制的一部分。也就是说，证据不是为了证明系统“做过某些检测”，而是为了证明系统的判断、执行与发布决策在语义上、治理上与运行上具有连续一致性。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

从系统结构上看，Evidence 的位置是被严格设计过的。当前设计在系统边界图中，将 **Evidence & Release Gating** 放在统一控制平面的右下方，并明确指出判断结果最终必须转化为 **release gating** 与 **audit artifact**。同时，在双轨评测框架中，系统要求两条评测轨道在右侧汇入 **combined decision board**，形成统一决策与证据输出；在风险驱动闭环中，系统进一步要求 **Measure → Profile → Select → Enforce** 之后接入 **Evidence & Release Gating**，再把其结果反向反馈给 **Measure**。这些结构性安排共同说明，Evidence 并不是系统边缘的“记录层”，而是控制平面将测量、判断与执行沉淀为治理对象的核心位置。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这一点同样重新定义了 **Release Gating** 的系统含义。当前材料已经明确指出，Evaluation 平台不仅服务于打分，更服务于 **release decision、责任边界与后续审计**；同样，Evidence 的存在也并不是为了补充解释，而是为了使判断结果能够进入真正的放行逻辑。由此，Release Gating 不应被理解为传统意义上的上线检查单，而应被理解为控制平面对“什么条件下允许进入下一阶段”所做的制度化裁决。它接收 benchmark 与 runtime judgment 所产生的证据，结合统一决策逻辑，对模型、控制配置或发布候选状态做出放行、限制或回退意义上的判断。也正是在这里，控制平面首次真正跨越“安全能力系统”的边界，成为“安全治理系统”。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

从方法论角度看，Evidence 与 Release Gating 的结合，使控制平面从线性执行系统升级为闭环治理系统。当前设计已经明确说明，评测、控制与执行不是一条单向流水线，而必须经由 **Evidence & Release Gating** 回流到后续的 **Measure**，形成 regression test 与持续改进机制。这意味着，系统并不满足于一次性地完成风险判断与运行时动作，而要求每一次执行都能够沉淀为后续 benchmark 与控制更新的输入。换言之，Evidence 的意义不仅在于支持当前的发布判断，也在于为未来的评测设计、控制策略与系统演进提供可追踪的反馈。正是在这一点上，Evidence 与 Release Gating 不再只是“记录”与“审批”，而是控制平面持续强化能力的关键结构。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

双轨评测结构进一步说明，Evidence 不可能是单一语义下的统一日志。当前设计已经明确指出，系统同时运行 **Global Safety & Robustness Track** 与 **China Regulatory Compliance Track（TC260）**，两者共享同一执行底座，但采用不同 **test suite、aggregation logic、report/output template**，最终合流到 **combined decision board**。这意味着，进入 Evidence & Release Gating 的并不是一类同质化结果，而是来自不同风险语义空间、不同聚合逻辑与不同报告模板的多源决策依据。由此，证据体系必须能够承载来自全球安全稳健性与中国监管合规性的不同判断结果，并在统一控制平面中把它们组织为可共同参与 release decision 的治理对象。也就是说，Evidence 的真正作用，不仅是存储结果，更是吸纳并协调多轨评测所产生的异质证据。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

同样地，Runtime Security Kernel 的定义也直接决定了 Evidence 的内容边界。当前设计已经明确指出，运行时层输出的不是简单的 allow / block，而是 **Scoring、Aggregation、Allow / Block / Redact / Degrade** 的综合判断；同时，xCloud 场景要求 **explainable blocking reasons**，系统已强调 **scoring system** 与 **explainability engine**。这说明，进入证据链的不应只是“发生了阻断”这一事件，而应包括判断是如何通过 scoring、aggregation 与 explainable reasoning 形成的，以及最终为何采取 allow、block、redact 或 degrade 等不同动作。由此，Evidence 体系必须天然兼容运行时的解释性输出，因为如果缺乏这一层，Release Gating 与 Auditability 就只能基于动作结果，而无法基于判断逻辑本身展开。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这也正是 **Auditability** 在本体系中被重新定义的原因。当前设计并未把 audit 理解为独立的合规附加功能，而是将其与 **Evidence & Release Gating** 一并纳入统一结构，并明确要求结果服务于 **audit artifact**。从蓝皮书的角度看，这意味着审计能力不再只是对历史日志的查看权限，而是对控制平面判断链条的可追踪性要求：benchmark 如何产生、judgment 如何形成、runtime 如何执行、release decision 如何得出，都必须在统一证据结构中保留可引用、可回溯的关系。当前设计虽然没有把这种关系形式化为更细粒度的数据结构，但它已经非常清楚地给出了最重要的原则：**判断不能脱离 benchmark，执行不能脱离 judgment，放行不能脱离 evidence，审计不能脱离整个链条。** [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

在统一研发框架中，Evidence 的地位同样被明确提升。当前设计在 “Unified R&D Framework” 页面中，已经把 **Evidence / CI-CD / Release Gating** 定义为与 **Security Science & Benchmark Governance、Runtime Security Kernel、Chinese & Multilingual Safety Intelligence** 并列的核心单元之一，并明确指出前四张图所形成的系统突破、评测突破、中文/合规突破与理论突破，不应由割裂团队分别推进，而应在统一 program 中协同完成。这个安排意味着，Evidence 与 Release Gating 不应再被视为交付后期的整合工作，而是从 program 设计之初就必须被视为一级研发对象。它的职责不是“兜底整合”，而是使评测、判断、执行与放行在组织和系统上真正闭环。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

正因为 Evidence 位于整个控制平面的治理中心，本章也必须明确系统**不应如何理解证据与放行体系**。首先，它不应被理解为运行日志归档，因为当前设计已经明确 Evidence 与 benchmark、judgment、runtime API 与 release gating 构成连续链条；其次，它不应被理解为单纯的报告输出，因为双轨评测已经说明不同轨道具有不同 report/output template，但最终都要进入统一决策与证据输出；最后，它也不应被理解为上线前的静态审批步骤，因为风险驱动闭环已经要求 Evidence & Release Gating 反向反馈到 Measure，成为 regression 与持续改进的一部分。只有在这些误解被排除之后，Evidence Graph、Release Gating 与 Auditability 才能被正确理解为控制平面的制度化中枢，而不是外围治理附件。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

综上所述，本章的核心结论是：**Evidence Graph, Release Gating, and Auditability 构成 Unified AI Security Control Plane 的治理闭环核心，它把 benchmark、judgment、runtime enforcement 与 release decision 连接为统一证据链，并通过统一决策输出、audit artifact 与回流到 Measure 的反馈机制，使控制平面具备可追踪、可放行、可审计与可持续强化的系统能力。** 当前设计已经明确给出了这一体系的结构基础与逻辑基础，包括 **benchmark → judgment → runtime API → evidence chain** 的统一能力、双轨评测的 **combined decision board**、风险驱动闭环中的 **Evidence & Release Gating** 反馈机制，以及 **Evidence / CI-CD / Release Gating** 作为统一研发 program 核心单元的组织定位。由此，Chapter 6 并不是蓝皮书中的管理附录，而是在方法论上正式确立：企业级 AI 安全系统必须能够把判断与执行转化为治理意义上的证据与放行逻辑，否则就无法真正成为控制平面。 [[统一 AI 安全控制平面 | PowerPoint]
