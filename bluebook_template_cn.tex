\begin{bluebookabstract}
本蓝皮书提出一套面向企业级 AI 部署的 \textbf{保证生产系统（guarantee production system）/义务闭合机器（obligation closure machine）}：在统一版本面下，把义务语义、运行时判断、强制动作、证据记录与放行陈述闭合为同一可管理对象，持续产出 \textbf{可部署、可审计、可回放、可收缩的 bounded guarantees}。控制面命名的，是在 \texttt{policy\_version} 与 \texttt{correlation\_id} 约束下贯通判断、强制与证据的最小系统边界，而非 detector 数量的堆叠。

为实现这一对象，本书采用“一主两翼”的结构。\textbf{主}为 \textbf{Unified AI Security Control Plane}，负责在统一边界内闭合 judgment、execution、evidence 与 release 四个基本对象；\textbf{翼 A} 为 \textbf{Decoupled Safety}，其作用不是增加一层安全外挂，而是把开放语义空间中不可被完全全局审查的问题，改写为事件可判定、策略可求值、失败可退化、证据可记录的实例级契约；\textbf{翼 B} 为 \textbf{Evaluation Framework}，其职责也不只是评分，而是把 benchmark constitution、acceptance contract、多轨评测纪律与机制证据包、放行证据包组织成同一证据链，使测量、回归与发布治理进入统一决策。三者的共同目标，不是构建“更大的安全平台”，而是使企业能够在明示前提下，对自身 AI 系统给出合法、可验证、可撤回的安全主张。

本书的核心判断是：企业级 AI 安全的根本问题不是“模型是否更对齐”，而是企业是否拥有一套能够持续闭合 judgment、execution、evidence 与 release 的控制系统。随着企业 AI 从单模型端点演化为包含网关、检索、工具、工作流、记忆与多轮状态的复合系统，风险已扩展为机密性、完整性、可用性、可控性、可审计性、可组合性与效用保持之间的系统性张力。若仍将安全寄托于更强模型、局部护栏或离线分数，义务就会在策略、线上行为、证据链与放行材料之间断裂。控制面的意义，正是在于把这些张力从“各团队局部优化”转化为可版本化 policy、可执行强制、可 join evidence 与可发布 bounded contract。

本蓝皮书给出的是\textbf{有界、部署导向的主张}，而不是对开放自然语言交互的全局安全证明。任何安全主张只有在其 scope、preconditions、degradation semantics、deployment regime 与 evidence basis 被明示，并能被编译到 acceptance contract 与 release evidence pack 时，才构成合法的企业级 guarantee statement。本书将 evaluation 写成 \textbf{语义分叉与治理汇合} 的证据引擎：不同法域、语言与义务空间在共享执行骨架上保持独立语义，并在统一治理决策板上汇合为 risk profile、release recommendation 与 audit artifact。企业级运行模型将 \textbf{theory、system、benchmark 与 release} 压成同一制品链路，使每次发布成为可辩护、可回放、可回滚的 bounded contract。

\end{bluebookabstract}

\executivesummary{
企业级 AI 安全的核心，不再是“如何为模型增加更多安全能力”，而是“如何持续生产、维护并辩护可签署的安全保证”。本蓝皮书因此提出的总对象，不是一个 AI 安全平台，也不是一套控制面模块，而是一台 \textbf{保证生产系统}：它要求企业在统一版本面下，将义务语义、运行时判断、强制动作、证据记录与 release 决策闭合为同一对象，使安全主张能够被生产、验证、收缩、回放并进入发布治理。Unified AI Security Control Plane 是这一对象在系统层的边界形式，而不是对它的全部解释。

这一总对象以“一主两翼”的方式被组织起来。主轴是 \textbf{Unified AI Security Control Plane}，其任务是在统一边界内闭合 judgment、execution、evidence 与 release 四个核心对象；第一翼是 \textbf{Decoupled Safety}，它将安全从模型内部难以稳定引用的隐式属性，改写为 policy、event、decision、evidence 与 Fail-Safe 对象上的实例级契约；第二翼是 \textbf{Evaluation Framework}，它把 benchmark constitution、acceptance contract、多轨评测纪律与两类 evidence pack 组织为同一证据引擎，使系统的测量、回归与放行答辩进入统一治理语法。由此，AI 安全不再被理解为“局部防御能力的叠加”，而被理解为一种能够持续产出 bounded guarantees 的企业系统。

本书之所以必须采用这一视角，是因为企业级 AI 部署已不再是单模型补全问题，而是一个由网关、检索、工具、工作流、记忆、多轮状态与审计流程共同构成的复合系统。风险也因此不再局限于有害输出，而扩展为机密性、完整性、可用性、可控性、可审计性、可组合性与效用保持之间的系统性张力。若仍将安全寄托于更大的模型、更多的离线对齐，或若干松耦合护栏的局部最优，组织将无法在 policy、线上行为、证据链与放行材料之间维持一致性。安全义务最终会断裂为孤立分数、孤立日志与孤立拦截，而无法成为可签署、可治理的 release object。

在系统原理上，本书提出的不是“再增加一层安全外挂”，而是一条必要路径：面对开放语义空间中无法被完全在线全局审查的问题，系统必须退到一个 \textbf{可计算外壳（computable envelope）} 上。Decoupled Safety 的意义，正在于把开放任务空间压缩为有限的 event、有限的 decision 语义、有限的 Fail-Safe 动作空间与有限的 evidence 字段，使热路径内的决策可以被确定性求值、被写入证据、并在冲突时进入协议化收缩。由此，运行时不再依赖模型在压力情境下的自我反思，而是由 policy、kernel、FailSafeObject 与 evidence graph 共同承担可执行的安全义务。

在方法论上，本书将企业级 AI 安全建模为一个 \textbf{双时间尺度的风险自适应控制系统}。快时间尺度位于在线热路径：系统在单请求、单会话预算内完成 event 求值、动作派发、Fail-Safe 收敛与证据追加；慢时间尺度位于离线冷路径：系统通过 benchmark、risk profile、multi-track evaluation、release gating 与 regression 持续更新 policy\_version、检测器包、能力包与放行边界。Evaluation 在此不是评分器，而是 \textbf{decision-signal compiler}；Evidence 与 Release Gating 也不是末端归档，而是把判断与执行提升为可放行、可审计、可回归的保证对象。

在评测与治理层，本书拒绝把不同风险语义压缩为单一总分，而采用 \textbf{semantic bifurcation with governance convergence} 的结构。全球安全稳健性、法域合规性、语言特定风险与组织义务在共享执行骨架上并行存在，各自保留 suite、aggregation logic 与 report semantics；它们最终汇入统一治理决策板，输出的是放行、限权放行、灰度或回滚等 release decision，而不是被隐式压缩为一个分数。因此，Track A 与 Track B 的纪律是本书的硬约束：Track A 承载可部署结论，Track B 承载机制上界与压力验证，二者不得被混读或混算。所有对外与对内可陈述保证，都必须由 Mechanism Evidence Pack 与 Release Evidence Pack 支撑。

本书同时明确给出一套 \textbf{claim discipline / guarantee grammar}。任何允许进入 release defense、对外材料或合同文本的安全主张，必须同时声明其 scope、invariants、degradation semantics、preconditions、evidence basis 与 deployment regime；任何无法被 acceptance contract 与 evidence pack 编译的陈述，都不被视为合法 guarantee statement。由此，本书提供的不是全局自然语言安全证明，而是一套 bounded deployment contract：在明示前提成立时，系统可以主张哪些有界保证；当前提破坏时，系统又必须退入何种最低服务形态，并同步收缩对外主张。

最后，本蓝皮书将 unified R\&D framework 视为 operating program（统一制品链路与节律的约定）：\textbf{theory → system → benchmark → release} 的单链路。细节见附录。

\section{战略背景与企业级转型}
\textbf{主线：}\ControlPlane{}。随着企业级 AI 从单一模型端点演化为包含网关、向量与文档索引、工作流、工具调用、人机环节与多轮状态的复合系统，风险也从“输出是否有害”扩展为机密性、完整性、可用性、可控性与治理可辩护性的系统性问题。提示注入可以穿越信任边界，检索投毒与隐指令可以进入上下文，工具可以把语言映射为高副作用行为，编排状态可以在轮次间累积，而一旦日志与决策无法共享同一关联对象，问责链与放行链就会同步断裂。正是在这种背景下，传统的应用安全外挂、内容安全模块或单模型对齐增强，已经不足以承担企业义务：它们可以优化局部 KPI，却无法在统一版本面上给出同一套可版本化策略、可执行强制与可回放证据。

\section{系统边界与责任划分}
\textbf{主线：}\ControlPlane{}。本书首先固定两类责任边界：其一是\emph{安全判断责任}——策略解释、风险评分、允许/拒绝/改写、理由码与结构化证据；其二是\emph{执行平台责任}——网络、扩缩容、编排 \textsc{sla}、算力与负载放置。\ControlPlane{} 的语义锚点属于前者，而不属于后者：它既不替执行平台承担编排责任，也不把安全问题降格为“换更大模型”或“增加更多外挂组件”。凡进入提示边界的外部对象——文档、检索块、工具契约、\textsc{api} 响应——都应被视为攻击面的一部分，并在同一 \texttt{policy\_version}、同一事件语义与同一 evidence schema 下被编目和约束。

\noindent\textbf{耦合的 \textsc{sla} 面。}对同步时延的保证（第4章 S1）同时依赖执行平台预算与同一请求包络内的控制面求值、证据追加预算；任一侧突破预算时，对外可契约化陈述须与第5章验收条款一致地收缩，而非将责任推给「另一子系统」。

\begin{problemstatement}
本书的问题陈述是：在动态语境、检索、工具、副作用、记忆与多租户约束下，\textbf{judgment、execution、evidence 与 release} 是否能在同一版本面与同一关联键下闭合。若四者不同框，则系统即便局部“安全”，也无法形成企业级可辩护保证：线上生效策略与放行材料不一致，审计无法复原当时依据的策略与检索/工具事件，回归与生产不共享同一事件与证据对象。\textbf{结论因此可以被压缩为：}在本书语境下，\ControlPlane{} 是\textbf{实现义务闭合机器的最小工程边界}——将 judgment 编译为 execution、将 execution 沉淀为 evidence、再将 evidence 编译为 release。
\end{problemstatement}

\section{控制面必要性：义务—断裂点—总对象}\label{sec:why-control-plane-necessary}
\textbf{主线：}\ControlPlane{}。

\subsection{义务闭合的必要性草图（形式化叙述）}
企业对外可陈述的保证，仅当在同一版本面上 obligation 语义、运行时 judgment、强制执行、证据沉淀与放行陈述可被对同一请求相互重构时才成立——否则局部「安全」无法\textbf{组合}成可审计的系统级保证。若标识、schema、轨道彼此割裂，则谈不上 \emph{义务闭合}：不存在可签署 bounded contract 的联合对象。本书将控制面作为最小工程边界，正因为唯有此处强制共享 \texttt{policy\_version}、事件类型与关联键——而非堆叠更多检测器。

\noindent\textbf{三类义务断裂（无控制面则闭合失败）。}第一，\textbf{租户与数据义务}无法被单一权重稳定表达；不同租户、法域与数据分级要求义务必须落在可版本化 policy 与可审计绑定上，否则模型升级会静默改写有效策略。第二，\textbf{检索与工具义务}附着在检索切片、索引世代、工具能力矩阵与审批路径上；若没有统一事件面，发布时就无法复原“系统当时允许了哪段外部文本、哪次高副作用调用”。第三，\textbf{放行义务}要求 release 材料必须能引用与线上一致的策略包、模型路由、检测器包与必要遥测样本；若证据 schema 不统一，则法务、数据治理与内部问责无法共享同一放行对象。唯有控制面形态的边界能够在同一编目与版本面下把义务定义、在线求值、证据沉淀与放行答辩收敛为一个闭合对象。

\section{边界刻画与四元闭合判据}\label{sec:boundary-characterization}
\textbf{主线：}\ControlPlane{}。本书将 \ControlPlane{} 视为这样一个边界：在其内，\textbf{judgment、execution、evidence、release} 四元必须共享 \texttt{policy\_version}、相容的事件 schema 与关联键，使义务在动态条件下仍可相互重构（见第\ref{sec:why-control-plane-necessary} 节）。凡将标识、schema 或轨道割裂的架构，若不经统一和解，则难以单独维持有界部署合同。

\noindent\textbf{可检验闭合判据（工程口径，非元数学公理）。}(i)~凡受治理的同步决策须引用 \texttt{policy\_version}，且在验收契约要求下留下可 join 的 \texttt{decision\_record\_id}。(ii)~同一请求在三平面上不得出现无显式和解或降级路径的标识冲突。(iii)~放行材料与线上强制所声称的范围须引用同一版本面。

\section{非目标与范围外架构}
\textbf{主线：}\ControlPlane{}。\emph{推论}（与第\ref{sec:boundary-characterization} 节的边界判据衔接）：仅以生成侧 \textsc{sla} 为优化目标的模型中心栈若缺少租户/检索/放行回归面；或互不关联的安全能力堆叠且无共享事件/证据标识，均无法单独维持有界部署合同。唯有控制面形态的强制边界才要求线上与离线在同一 \texttt{policy\_version} 下可对齐，并以统一证据对象支撑差分重跑。

\section{阅读约定（摘要）}
\textbf{主线：}\ControlPlane{} / \EvalFramework{}。本书在明示前提下（第5章）承诺提供\emph{可部署、可测量、可撤回}的契约；非目标见上文。benchmark governance 定义测量条件；control plane 承担 judgment 与策略求值；执行环境承担 \textsc{sla} 而不承载安全语义；证据/放行闸与法域化语义（第6章）形成可放行对象。

\section{保证生产的工程规则：版本面与 join 条件}
\textbf{主线：}\ControlPlane{}。\textbf{工程规则：}一条可对外陈述的 bounded guarantee，必须能在证据面被可 join 地复原。最低要求包括：\texttt{policy\_version}、\texttt{correlation\_id}、\texttt{decision\_record\_id}、\texttt{rationale\_code}，以及（若触发 Tier-R2）\texttt{approval\_ref}。缺任一键，则放行材料无法证明“当时依据什么、执行了什么、记录了什么”，不构成可辩护保证对象。

\section{资产、对手模型与运行假设}
\textbf{主线：}\ControlPlane{} / \EvalFramework{}。受保护资产包括：用户与企业数据的机密性，提示与检索内容的完整性，工具侧效果与参数完整性与不可否认性，服务在 \textsc{sla} 内的可用性，以及审计记录的完整性。考虑的对手包括：构造对抗性提示；经检索信道投毒或泄露；滥用或串联工具调用；操纵会话与工作流状态；滥用运营面（如文档中的隐匿指令）。内部威胁主要通过职责分离、证据不可篡改与工具最小权限来缓解——而非仅靠模型对齐。

运行假设包括：策略可以机器可核验的片段表述；在网关、检索、推理、工具与日志层存在强制挂点；部分残余风险通过\emph{验收契约}明示接受；非确定性须由统计检验计划与在线监控界定，而不能被笼统忽略。

\paragraph{控制面算力与排队枯竭。}除错 join 外，对抗流量、深层检测级联或证据总线背压亦可耗尽控制面 CPU、内存或追加配额——须作为一等威胁建模：租户级预算、tier 超时与 skip 规则（第4章）、证据路径准入控制，以及在截止压力下进入保守 D 动作，而非静默绕过策略求值。

\section{安全目标与权衡}
\textbf{主线：}\ControlPlane{} / \EvalFramework{}。下列目标在系统层面须一并满足；每条均附\emph{若不纳入控制面则必然失灵}的判据，以钉死总对象必要性。

\begin{securityobjectives}
\begin{itemize}
  \item \textbf{机密性（Confidentiality）。} 敏感数据不得经提示、检索、日志或工具输出逾越策略信任边界。\textbf{若不在控制面：}各模块独立脱敏，无法证明「同一次跨检索—工具链」未泄漏切片级义务。
  \item \textbf{完整性（Integrity）。} 提示、检索片段、工具参数与侧效果在威胁假设内可检。\textbf{若不在控制面：}放行与线上求值使用不同字段定义，完整性主张不可辩护。
  \item \textbf{可用性（Availability）。} 控制在 \textsc{sla} 内可服务；降级可度量。\textbf{若不在控制面：}安全与平台各报各的延迟，无法对「带安全时的 p99」签单一合同。
  \item \textbf{可控性（Controllability）。} 高风险行为由显式运行时策略约束。\textbf{若不在控制面：}「模型习性」无法版本化，亦无法进入回归闸。
  \item \textbf{可审计性（Auditability）。} 策略相关决策产生可回放结构化证据。\textbf{若不在控制面：}审计与生产日志无法共享关联 ID，事故复盘止于猜测。
  \item \textbf{可组合性（Composability）。} 控制在异构模型与栈上仍有效。\textbf{若不在控制面：}每次换模型即换一套隐式策略，组织无法积累可比较证据。
  \item \textbf{效用保持（Utility Preservation）。} 无谓损害最小化；无谓拦截为缺陷指标。\textbf{若不在控制面：}局部模块各自优化拦截率，全局效用无人负责。
\end{itemize}
\end{securityobjectives}

\noindent\textbf{权衡。}机密性/完整性与效用、可用性常冲突；审计深度与成本、时延冲突。此类张力必须在控制面内\emph{政策化}（版本、阈值、例外与证据包），否则沦为各团队即兴谈判。\DecoupledSafety{}（第2章）提供外置策略优先机制；\EvalFramework{}（第5章）提供可测与可放行证据；二者均依赖本章确立的总对象边界。

\designprinciple{解耦（支撑 judgment）}{策略与模型权重分离，使「当时依据哪条规则」可被引用与签字。}
\designprinciple{跨层强制（支撑 execution）}{从输入、检索、推理、工具到输出与编排日志的强制挂点处于同一事件与版本语义下。}
\designprinciple{证据先于信任（贯通 evidence 与 release）}{任何对外与对内保证须可指向测试、遥测或签批制品，并与放行对象同一版本面。}

\keytakeaway{企业级 AI 安全的中心矛盾是四元是否闭合：判断、执行、证据与放行必须在 \ControlPlane{} 的同一编目与版本面下可关联、可差分重跑；否则在复合系统下义务必然断裂——故总对象为 \ControlPlane{} 这一承载共享版本与 join 语义的最小边界；单一模型平台或无关联能力堆叠本身不提供该闭合。}

\section{必要路径：从开放语义到可计算外壳}
\textbf{形式化外壳（有限基）。}记有限集合：事件类型 \(\mathcal{E}\)、动作 \(\mathcal{A}\)、理由码 \(\mathcal{R}\)、策略制品版本 \(\mathcal{V}\)。热路径语义为 \(\mathcal{E}\times\mathcal{V}\rightarrow \mathcal{A}\times\mathcal{R}\) 且受预算约束；超出载体的语义一律带外直至冷路径版本抬升。

企业级 AI 安全的首要难题，不是“如何再提高一点模型的安全倾向”，而是：在开放语义空间中，系统不可能在热路径内对所有语境做全局、完备且低成本的安全判定。开放任务、多轮状态、检索注入、工具副作用与法域义务共同作用，使“对一切语义做完全在线审查”在工程上不可执行，也无法形成企业可签署的稳定合同。可部署的替代路径，因而不是继续追求全局语义审查，而是退到\textbf{可计算外壳（computable envelope）}：系统不要求在运行时理解全部自然语言含义，而要求把开放空间投影为有限的事件类型、有限的动作集合、有限的退化协议与有限的证据字段，使热路径内的决策可以被确定性求值、被写入 evidence、并在必要时触发 fail-safe 收缩。

从这一点出发，\DecoupledSafety{} 的真正工程含义就变得清晰：它不是“把安全从模型里拿出来”这么简单，而是把安全从不可签署的开放语义愿望，改写为可以进入 \texttt{policy\_version}、\texttt{event} schema、\texttt{decision} 语义与 evidence graph 的实例级契约。热路径承担的是\textbf{在有限预算内完成求值、派发与记证}；冷路径承担的是\textbf{以版本化制品更新控制律，而不在线上发生隐性语义突变}。这也是本书后续所有 runtime protocol、evaluation evidence pack 与 bounded deployment contract 的系统前提。

\paragraph{投影映射与残余漏报。}开放语义语境经规范化与离散分桶映射到 \(\mathcal{E}\) 的投影 \(\pi\) 必然\textbf{有损}：无法区分为可判别 \texttt{event} 特征的对抗或义务会变成热路径边界上的\textbf{漏报（false negative）}。单靠堆叠检测器无法消灭该类误差；必须由冷路径界定：Track B 压力套件、ORA 驱动的检测器与策略制品更新，以及 acceptance contract 中显式承担残余风险的切片（第5章）。\textbf{量化咬合：}与 \(\pi\) 关联的\textbf{损失率}由 Track B 压力套件中的漏检率、场景覆盖等指标度量，并进入第5章 acceptance contract 的误差预算，用以区分可部署陈述与上界/统计陈述。

\paragraph{工程语义 vs.\ 形式证明。}本章有限集合 \(\mathcal{E},\mathcal{A},\mathcal{R},\mathcal{V}\) 及映射表述的是\textbf{可签约的工程语义}——实施与验收应对齐的对象；并非对开放式 AI 系统完备形式正确性的证明。

\section{引论：对齐与系统安全的间隙}
监督微调、偏好对齐与安全后训练可降低孤立补全场景下的有害输出概率，但并不自动推出含租户义务、检索血缘、工具与放行约束的企业级系统安全（第1章）。\DecoupledSafety{} 是对齐的\textbf{补完}而非替代：生成仍在模型侧；policy、Fail-Safe 协议、证据 schema 与放行逻辑必须外置为稳定、可版本对象。

\section{三类基础失效及其设计含义}
\textbf{第一类失效，是语义审查失效与计算—验证不对称。}在开放任务与富语境条件下，把“审查所有有害语义”设为统一在线目标，会立刻遭遇可执行性、算力预算、时延与测试口径的冲突。其设计含义不是“再上更强审查器”，而是把验证收缩到实例级对象：输入块、检索单元、工具调用、候选输出等离散 \texttt{event} 必须在版本化 \texttt{policy} 下被可重复求值，并由 \texttt{kernel} 产出带理由码的 \texttt{decision} 与可 join 的 \texttt{evidence}。

\textbf{第二类失效，是认知一致性失效与价值/本体锚定的必要性。}多轮对话、社会技术压力与工具链副作用会破坏“模型会在更深思考后自行保持安全”的假设。其设计含义是：价值边界、本体约束与可接受动作空间必须被写入外置 \texttt{policy}；一旦冲突信号成立，系统必须触发 \texttt{FailSafeObject} 所定义的受控退化，而不是继续开放式生成；恢复必须是被授权、可记证、可引用的恢复。

\textbf{第三类失效，是文本语义检索路径失效与表示级/行为级监控的必要性。}仅靠表面 token 或局部 prompt 语义无法覆盖检索投毒、隐性指令、跨文档拼接与排序操纵。其设计含义是：检索侧必须强制血缘、分段与引用绑定；监控与证据中必须保留表示级或行为轨迹级信号接口，使“系统当时看见了什么、为何判定异常、为何进入退化或限权”可以进入第5章的评测与证据体系。

\section{最小对象模型（与第3--5章接口）}
为避免 \DecoupledSafety{} 退化为原则性语言，本书固定以下最小对象集合，并要求贯穿 Chapter 3--5 的实现、协议与证据体系。\texttt{policy} 表示版本化义务、阈值与允许动作空间；\texttt{kernel} 消费 \texttt{event} 产出 \texttt{decision} 并发射 \texttt{evidence}；\texttt{environment} 描述拓扑、信任域、观测预算与部署分层；\texttt{event} 是输入块、检索命中、工具调用、候选输出等类型化安全事件；\texttt{decision} 是 allow/rewrite/sandbox/escalate/block/degrade 等有限动作及理由码；\texttt{evidence} 是可关联、可回放、可差分的记录集合；\texttt{FailSafeObject} 将触发条件、允许退化集合、服务不变量与恢复层级绑定为同一协议对象；若系统启用带外适应，则 \texttt{OOB\_adaptation} 描述观察、路由、适应与上线的冷路径配置及其证据锚。

\section{典型失效模式}
\securityrisk{归属模糊}{安全逻辑隐匿在 prompt 技巧或模型权重中时，升级会在无回归对象的情况下静默改写有效策略。}
\securityrisk{编排绕检}{智能体与工作流通过备用工具或隐式信道绕开强制，导致控制一致性崩溃。}
\securityrisk{证据缺失}{缺少稳定 evidence graph 时，组织无法复原“为什么当时允许/拒绝、依据哪版 policy、哪次检索/工具”。}

\section{有界主张与本书安全契约（bounded claim）}
\DecoupledSafety{} 在本书中的承诺是有界且可撤回的：它不承诺对任意自然语言任务与任意工具生态给出全局、完备、可判定的安全；也不承诺模型永不犯错。本书所承诺的是：\textbf{在 environment 明示、policy 与检测器版本可查、观测与算力预算可被度量、acceptance contract 已签入的前提下}，系统能够提高攻击者成本、限制失效半径，并在前提被破坏时退入预声明的最低安全服务形态。

\researchagenda{基础问题}{在异构模型、非确定解码与多步工具使用下，哪些组合式约束能够在可测误差界内被强制执行，同时其 evidence pack 仍满足可部署轨（Track A）的放行纪律？}

\keytakeaway{\DecoupledSafety{} 在本书中不是安全外挂，而是面对开放语义、外部检索、工具副作用与多轮状态时，把安全改写为“可计算外壳 + 实例契约 + Fail-Safe 协议 + 可 join 证据”的系统原理；它的价值不在于提供全局证明，而在于使企业级 AI 安全第一次具备被生产、被维护、被审计和被发布辩护的可能。}

\section{责任拓扑到 join 拓扑：系统边界如何被证据闭合}
\textbf{主线：}\ControlPlane{}。第1章将控制面定义为责任边界拓扑；在系统层面，这一拓扑最终必须落成\textbf{join 拓扑}：哪些键必须在数据面/控制面/证据面同时出现，才能把一次 judgment 与其 execution、evidence、release 复原为同一叙事。故本章的“参考架构”不以组件数量为胜，而以\textbf{闭合能力}为胜：同一 \texttt{policy\_version} 约束热路径求值与冷路径聚合；同一 \texttt{correlation\_id} 串联事件序；同一 \texttt{decision\_record\_id} 使证据可被发布闸引用。缺任一键，则控制面退化为平台装饰，无法承担义务闭合。

\section{双平面正确性：热路径原子强制与冷路径版本单调}
\textbf{主线：}\ControlPlane{}。双平面切分不仅是性能优化，而是正确性条件：\textbf{热路径}必须在 S1 预算内完成确定性求值、受控派发与证据原子追加；\textbf{冷路径}只能通过版本化制品（\texttt{policy\_version}、检测器包、索引世代、路由表）更新控制律，并经发布闸进入热路径。任何绕开版本单调的“线上即兴改动”都会破坏可回放与可辩护性，直接击穿 guarantee production system 的核心承诺。

\section{分层参考架构与动态控制流}
\textbf{主线：}\ControlPlane{}。分层描述回答「谁负责什么」；\emph{动态控制流}回答「一次请求中状态如何迁移」。二者不可缺一：交互—语境—模型—运行时—治理各层在运行时以\textbf{事件}串联——同一 \texttt{correlation\_id} 下依次产生语境绑定、检索命中记录、模型路由决策、\SafetyKernel{} 求值与证据追加。静态图若不与事件序对齐，工程上会出现「图画了、日志对不上」的断裂。

\section{数据面、控制面与证据面：最小字段}
\textbf{主线：}\ControlPlane{}。三平面不是哲学标签，而是\emph{写入与校验的最小列集合}；下列字段为推荐最小集，实现可增列但不得拆散关联键。

\noindent\textbf{关联规则。}任一 \textbf{decision} 必须能由（数据面子集 + 控制面子集）在证据面中唯一重构；发布闸与回归只承认满足该 join 闭合的记录（与第5章最小证据包一致）。

\subsection{Join 作为主不变量：组合证据语法（compositional grammar）}
Join 不应仅被当作文档粘合，而应作为\textbf{主系统不变量}：在同一 \texttt{correlation\_id} 下，组件级证据必须在同一 \texttt{policy\_version} 下组合为单一可回放叙事。记网关/输入绑定、检索血缘、生成与内核判定、工具副作用对应的组件记录为 $E_{\mathrm{in}}, E_{\mathrm{ret}}, E_{\mathrm{gen}}, E_{\mathrm{tool}}$。系统级主张采用在同一套标识下的组合 join（\texttt{rationale\_code} 语义沿链相容）：
\[
  G_{\mathrm{sys}} \equiv E_{\mathrm{in}} \oplus E_{\mathrm{ret}} \oplus E_{\mathrm{gen}} \oplus E_{\mathrm{tool}} \quad\text{（共享 \texttt{correlation\_id}、\texttt{policy\_version}）.}
\]
\textbf{合法组合：}各节点引用同一 \texttt{policy\_version}；证据行按声明键 join；理由码沿链不自相矛盾。\textbf{非法组合（shadow bypass）：}生成证据宣称「输出安全」但同一请求在高副作用工具调用上缺失 $E_{\mathrm{tool}}$；或检索血缘缺失却敏感切片已进入提示——即便局部分数好看，Track~A 的系统级结论亦应无效。

\noindent\textbf{工程闭合原则（组合证据，非形式化表述）。}当且仅当对某 \texttt{correlation\_id} 的各 Mandatory 子组件记录在声明的同步预算 S1 内齐备、可 join 且语义一致时，系统级可部署主张才\emph{可辩护}；缺失或错配 join 则否定主张，无论局部指标如何。

\section{可 join 证据总线：故障、分区与保证—可用性张力}
\textbf{主线：}\ControlPlane{}。组合语法 $G_{\mathrm{sys}}$ 假设键一致的可靠追加；分布式现实要求显式模式：\textbf{（i）证据路径退化}——若追加超 S1，同步接口仍须满足第4章 Fail-Safe；需完整 join 的主张必须收缩而非假装。\textbf{（ii）日志分区/不可用}——缺 mandatory $E_{\mathrm{tool}}$ 且确有副作用时，Track A「安全」叙述 fail-closed；除非 acceptance contract 明示弱观测区仅允许较弱保证（fail-open 条款）。\textbf{（iii）\texttt{decision\_record\_id} 签发失败}——不得在无治理记录下假装受控执行。

\paragraph{哈希链不可篡改与 S1。}若 \texttt{decision\_record\_id} 或追加路径采用哈希链/仅追加存储，\textbf{链的耐久扩展}（复制、跨分区排序）属于异步关切：热路径仍须在 S1 内返回稳定记录句柄及（若采用）哈希原像或日志序令牌；完整哈希校验与回填不得阻塞同步响应体，应与用户可见 \textsc{sla} 分离预算；若仅快路径追加可用，则按第4章 Fail-Safe 收缩保证口径。

\paragraph{典型失效模式（跨租户工具升级经由检索）。}对手污染检索使模型提议将数据外泄至越权目的地的工具调用。若检索证据 $E_{\mathrm{ret}}$ 未将切片绑定至 \texttt{decision\_record\_id}，或已发生调用却缺失 $E_{\mathrm{tool}}$，则 Release Evidence Pack 必须在准入规则下因\textbf{证据不完整}而失败——禁止仅凭生成侧分数构造「总体安全」叙事。

\noindent\textbf{join 完备性与运行模式。}\textbf{完备 join 是 Track A 可部署闭合的前提，}不等于「每种降级可用性状态下都必须达成」：同步接口可在更弱契约档次下仍可响应。\textbf{强制证据桥：}若 mandatory 行缺失而执行仍在继续，则仅能发生在验收契约\textbf{明示弱化保证}的档次；Track A「安全」叙事与强放行条款\textbf{暂停}直至回放补齐或范围收缩（第5章准入）。

\section{同步在线强制平面与异步带外适应平面}
\textbf{主线：}\ControlPlane{} + \DecoupledSafety{}。

\paragraph{同步在线强制平面（热路径）。}在单请求/单会话 \textsc{sla} 内必须完成：事件规范化、策略绑定、确定性求值、允许的动作派发、同步 \textsc{api} 可返回的结果、以及证据条目的\emph{原子追加}。延迟预算内允许的工作：轻量规则、缓存型分类器、限次检索、已预热的重检测路径。原则：\textbf{不依赖}跨请求全局优化、不阻塞于人工审批（除非已进入协议化 Tier-R2 且仍满足 S1）。

\paragraph{异步带外适应平面（冷路径）。}在请求边界之外运行：语料与索引刷新、跨租户漂移监控聚合、策略候选挖掘、评测回归全量重跑、红队战役、阈值与路由策略的批量调参、模型路由 A/B 的离线评估。产出物为\emph{下一版} \texttt{policy\_version}、检测器包、索引世代或路由表——经发布闸后才能在热路径生效。

\paragraph{切分纪律。}热路径只消费\emph{已发布}配置版本；冷路径禁止直接改写正在服务中的热配置而不经版本单调与签批。第4章 Fail-Safe 的触发信号主要来自热路径；适应动作的主体在冷路径完成。

\section{Controls-core：\SafetyKernel{} 的工程实例化（建议新增）}
\textbf{主线：}\DecoupledSafety{}。当前 controls-core 子模块表明，\SafetyKernel{} 可以首先以一个最小工程实例出现，而不必一开始就展开为完整控制平面。其核心抽象已被收敛为 \textbf{types \& interfaces、detection、scoring、prevention} 四类对象，并通过 \textbf{controls-core-lite / controls-core-advanced / controls-core-full} 三档构建，以及 \textbf{controls-local-service / controls-cloud-service} 两类部署承载，形成能力包与 deployment regime 的基础分层。换言之，kernel 的工程实例并不以“拥有多少 detector”为定义，而以“是否存在稳定的事件接口、可分层的执行 tier、可切换的能力包和可治理的服务承载方式”为定义。

从本书对象语言看，这一设计正对应于 kernel 的最小工程化：types \& interfaces 对应 event contract 与 detector contract，detection / scoring / prevention 对应 decision synthesis 与受控动作生成，lite / advanced / full 对应能力包分层，local / cloud service 则对应 deployment regime 的具体承载形式。更重要的是，该子模块已明确支持 \textbf{registry 自动发现、配置启停、API / worker 承接、DB / object storage / logging / post-analysis} 等机制，这说明 kernel 并不是一组静态规则，而是一个可以被版本化、被部署裁剪、被证据化并被后续 governance 引用的运行时 artifact。由此，controls-core 不是整个控制面的替代物，但它提供了一个可以直接映射到本章 kernel 接口、双平面切分与第5章 bounded deployment contract 的工程实例。

\section{可计算外壳的运行时实例}
\textbf{主线：}\DecoupledSafety{}。第2章给出\textbf{可计算外壳}的必要性；本章固定其\textbf{运行时实例}：有限触发（T1--T3）、有限动作（D1--D4）、不变量 S1、恢复层（Tier-R1/Tier-R2）及每次决策的可 join 证据字段——不是检测器枚举，而是可写入证据包的标记化协议。

所谓分层级联可由\emph{时延受限的运行时内核}实现；工程锚点（controls-core）见第3章 Controls-core 小节。

\section{在线热路径与带外冷路径：分工回顾}
\textbf{主线：}\ControlPlane{}。第3章已固定双平面正确性与预算；本章在预算内实例化 T1--T3 与 D1--D4——下文 S1 预算拆分与嵌套超时是对该切分的量化延伸。

\section{风险自适应控制闭环}
\textbf{主线：}\ControlPlane{}。在线强制与保证闭环对齐：
\begin{enumerate}
  \item \textbf{测量（Measure）。}离线/在线测量产生决策信号（攻击成功率、漂移、拒绝质量、时延等）。
  \item \textbf{画像（Profile）。}按模型与场景汇总风险暴露，给定部署语境。
  \item \textbf{选择（Select）。}在风险、成本、时延与业务容忍度约束下选择控制深度（规则对轻量检测对重型检测；工具约束；检索分段）。
  \item \textbf{强制（Enforce）。}经由 \SafetyKernel{} 接口确定性施加动作。
  \item \textbf{证据与反馈。}发射审计记录；失效驱动基准更新、红队优先级与策略修订。
\end{enumerate}

\section{策略决策生命周期}
\textbf{主线：}\DecoupledSafety{}。单次请求维度：
\begin{enumerate}
  \item 审视请求并提取风险信号（主题敏感度、信道信任、检索范围、工具暴露）。
  \item 将请求绑定到适用的策略包与版本（租户、地理、负载类别）。
  \item 评估干预：允许、改写、约束、沙箱、升级、阻断、降级，并给出显式理由码。
  \item 记录决策及可回放的相关证据载荷。
  \item 将事件与未遂注入离线套件与影子评测。
\end{enumerate}

\section{检索安全（\RAG{}）}
\textbf{主线：}\ControlPlane{}。检索将第三方文本带入提示边界；应将检索片段视为不可信、类代码内容。控制包括：语料分段（公开与机密）、元数据与血缘要求、检索时过滤、与段落绑定的引用、矛盾处理、自敏感切片导出答案的最大特权范围、以及批量与在线投毒检测挂钩。检索策略须与包含投毒文档场景的评测协同演化。

\section{工具调用控制}
\textbf{主线：}\ControlPlane{}。工具将语言映射为副作用；须按租户与工作流执行能力矩阵、参数类型校验、高影响动作的动态审批、可行时的沙箱执行、持久限速，以及只读与变更类工具的分离。编排器不得构造绕过已记录决策的平行路径。

\section{两阶段 Fail-Safe 服务协议（蓝皮书压缩写法）}
\textbf{主线：}\DecoupledSafety{}。本节把论文式触发—动作—不变量—恢复压缩为\emph{可验收叙述}：实现可保留内部编号，对外文档只需「阶段—触发类—动作类—不变量—恢复层」五元一致。\textbf{阶段一}在 \SafetyKernel{} 内完成检测与判定；\textbf{阶段二}在预声明降级空间内完成对用户可见的服务形态迁移，并写证据；恢复可跨至冷路径但须满足权限与版本纪律。

\subsection{触发类 T1--T3（热路径可读）}
\begin{itemize}
  \item \textbf{T1（策略/规范冲突类）}——规则互斥、越权工具意图、策略版本与运行态不一致、或义务在当前 \texttt{event} 上不可同时满足。热路径内须给出明确 \texttt{rationale\_code}；若无法局部消解则进入阶段二。
  \item \textbf{T2（社会技术/运营异常类）}——速率/来源/租户上下文突变、依赖服务降级、审批队列逼近或突破 \textsc{sla}。允许在热路径先触发「限流/只读/收窄工具面」等\emph{低成本}动作；若仍不满足 S1，则进入阶段二 D 类。
  \item \textbf{T3（验证器不确定/证据不足类）}——分类器低置信、检索血缘缺失、关键断言无法绑定证据面字段。默认倾向保守：进入 D1/D3 或 D4，而非在热路径上「猜过去」。
\end{itemize}

\noindent\textbf{有限状态视角。}实现应保证 T1--T3 覆盖热路径状态空间而无「未定义」结局：每个规范化 \texttt{event} 落入标注的触发类（可多信号映射到同类），再在阶段一给出 \SafetyKernel{} 内可消解的判定或带明确 \texttt{rationale\_code} 移交阶段二。不确定性走保守 D 动作或 Tier-R，而非静默默认值——使 Fail-Safe 层成为与证据字段对齐的\textbf{有限标记转移系统}。

\subsection{降级动作空间 D1--D4（受控四象）}
触发成立时，仅允许在下列\emph{预配置}动作间迁移；每项须带业务影响标签与必填证据列（与第5章 E1--E3 对齐）：
\begin{itemize}
  \item \textbf{D1}——模板化拒绝或固定安全答复（可预测、可测 Utility 代价）。
  \item \textbf{D2}——受限无状态沙箱：禁用或缩小高副作用工具集、收窄检索切片或索引世代。
  \item \textbf{D3}——对模型输出做安全删改/脱敏后返回（保留可引用理由码）。
  \item \textbf{D4}——硬终止当前请求：返回错误码与自助恢复提示，不隐式重试耗尽显性预算。
\end{itemize}

\subsection{同步服务不变量 S1}
\textbf{S1（同步可应答）：}凡声明为同步的 \textsc{api}，在含降级情形下仍须在约定时限内返回\emph{可机器处理的}结果体（含降级载荷或错误码），不得无限阻塞主链路；须返回与租户契约一致的理由码字段。S1 与第3章热路径预算一致：违反即视为实现缺陷而非「模型不够聪明」。\textbf{量化基线（依契约而定）：}交互类网关常声明端到端 $T_{\mathrm{budget}} \le 800\,\mathrm{ms}$（\textsc{p95}）等预算；各强制层须为其预留规范化、内核 tier 与证据追加的嵌套分片，否则须同步收缩 acceptance 范围。

\subsubsection*{S1 预算拆分与分层级联超时}
\textbf{硬性记账（非形式）。}记单次请求的同步包络为 $T_{\mathrm{budget}}$（如网关 \textsc{p95}），各分量须满足\textbf{可行性约束}
\[
  T_{\mathrm{norm}} + \sum_{k \in \mathrm{tiers}} T_k^{\mathrm{alloc}} + T_{\mathrm{dispatch}} + T_{\mathrm{append}} \;\le\; T_{\mathrm{budget}},
\]
其中 $T_{\mathrm{norm}}$ 覆盖解析与关联；tier $k$ 各有显式片 $T_k^{\mathrm{alloc}}$（层内并行仍消耗该层片）；$T_{\mathrm{dispatch}}$ 为受控下游动作；$T_{\mathrm{append}}$ 为治理执行所要求的同步证据追加完成时间。分层级联须嵌套截止：若某层超时，按 fail-fast 跳过或短路后续可裁量层、升级到保守 D 动作或拒绝——禁止单层阻塞拖垮整条路径导致被动 S1 违约。

\subsubsection*{有限标记转移示意（非形式 FSM）}
粗粒度状态 \(\{\mathrm{N},\mathrm{T},\mathrm{D},\mathrm{B}\}\)（正常、触发激活、D 域降级、阻塞/错误终态）。\textbf{基础设施级中断（分层）：}\texttt{decision\_record\_id} 签发失败、mandatory 证据行缺失、证据总线分区等\textbf{不作为}与 T1--T3 同层的普通矩阵输入；它们视为\textbf{全局中断}，优先于语义/策略 FSM，强制进入 \textbf{B} 或保守 \textbf{D} 并附 \texttt{rationale\_code}，不与「逻辑 T 状态」并发竞态。下表仅覆盖\textbf{语义/策略}信号。

\subsection{异构恢复 Tier-R1 / Tier-R2}
\begin{itemize}
  \item \textbf{Tier-R1（规则化恢复）}——清状态、回滚试验旗标、切换只读、切回较轻检测路径等自动化步骤；可在秒至分钟级完成；证据自动追加。\textbf{恢复窗口内的保证姿态：}在 Tier-R1 尚未收敛到可引用稳态前，同步流量相对 Track A 主张须\textbf{保证侧 fail-closed}——通常为保守拒绝或能力收缩——不得以旁路放行换取可用。
  \item \textbf{Tier-R2（人工/规范性恢复）}——法务、安全或数据治理签批后恢复全能力或提升特权；必须产生 \texttt{approval\_ref} 与策略版本单调记录；不得由运行时执行身份自批。
\end{itemize}
\textbf{authority boundary：}谁可触发 Tier-R2、可改哪些策略字段，与执行平台服务账号分离配置。

\section{带外受控适应闭环（ORA：Observe--Route--Adapt）}
\textbf{主线：}\ControlPlane{}。ORA 不是「让模型在线自学」，而是第3章\textbf{异步带外适应平面}上的\emph{受控闭环}：只消费已落盘的证据与聚合指标，产出\emph{下一版}可发布配置，再经签批进入热路径。

\paragraph{在线路径做什么。}在 \textsc{sla} 内：按当前 \texttt{policy\_version} 对 \texttt{event} 求值、派发、写证据；在 T1--T3 成立时进入 D1--D4 与 Tier-R1 可自动化部分。在线路径\textbf{不做}：全量重训、全索引重建、跨租户全局阈值搜索、未经签批的策略改写。

\paragraph{带外路径做什么。}消费 Observe 阶段汇总的监控与评测差分：漂移、误报/漏报结构、攻击面变化；在 Route 阶段生成候选（新阈值、检索分段策略、工具白名单变更、模型路由权重）；在 Adapt 阶段经离线评测与发布闸后\emph{版本化}上线。产出物永远是「新版本配置包」，而非热路径内存中的隐性突变。

\paragraph{可进入在线路径的信号。}单请求内可判定且预算内的：规则特征、缓存型分数、轻量检索校验、已预热重检测一次、与当前 \texttt{correlation\_id} 直接绑定的血缘缺失标记。其共同特征是：可在 S1 内完成求值且不需要跨请求全局最优。

\paragraph{必须异步完成的昂贵适应。}包括但不限于：大规模语料/索引刷新、跨租户聚合调参、完整红队战役回放、策略 DSL 的结构化改写、引入新检测器家族、改变 Track A 放行契约语义。此类动作只能在 ORA 冷路径完成，并在证据面产生「候选—评测—签批—生效」链，防止线上隐性分叉。

\paragraph{并发与 \texttt{policy\_version} 单调性。}多条 ORA 流可能并行提出重叠候选；须在 ship 前经版本控制式合并（分支—评审—合并）：保持对外生效 \texttt{policy\_version} 的\textbf{严格单调递增}，禁止冲突编辑不经发布闸覆盖线上有效策略——Git 式工作流为操作范式；无合并纪律的即兴并发在高负载下复现隐性分叉。

\section{语义分叉与治理汇合：semantic bifurcation with governance convergence}
\textbf{主线：}\EvalFramework{}。本书的方法论“控制论心脏”已在第4章（双时间尺度风险自适应控制）固定：热路径负责实例级强制与证据追加，冷路径负责版本级测量、回归与放行监督。本章因此不再重复命名该心脏，而只给出 \EvalFramework{} 在其中的角色：它不是输出抽象总分，而是把企业安全中多个不可通约的语义空间组织为\textbf{可放行引用}的证据制品。企业级 AI 安全并不存在单一评测语义：全球安全稳健性、法域合规性、语言特定风险与组织治理义务，往往在测试对象、聚合逻辑与放行语义上彼此不同。由此，本书中的多轨评测不应被理解为「多跑几套测试」，而应被理解为 semantic bifurcation：在共享执行骨架上保留不同语义轨道的独立性，并在放行时实现 governance convergence，即通过同一治理决策板、同一 acceptance contract 与同一 evidence pack 体系完成可部署结论的汇合。

这一结构的关键不在于压缩分数，而在于统一治理。统一治理决策统一的是「是否放行、以何种能力包放行、在何种 bounded contract 下放行」，而不是把不同语义轨道强行压缩为单一 score。凡需要跨轨压缩的地方，都必须显式披露其权重、损失与适用范围，并将其视为新的主张前提，而不能被当作天然存在的「总分事实」。因此，本章中的 evaluation 从一开始就不是测分工具，而是 guarantee production system 中的语义编译层：它决定系统可以在哪些语义空间中、以何种证据强度，对外主张何种保证。

\section{从测量到控制选择：评测输出作为决策信号（decision signals）}
\textbf{主线：}\EvalFramework{}。在本书中，评测的输出并不终止于报告，而是被编译为 \ControlPlane{} 与 release governance 可消费的\emph{decision signals}。它们回答的不是「哪个模型分更高」，而是：哪些场景需要升级到更强控制深度，哪些能力包必须被收缩，哪些法域或语言尚不足以进入可部署轨，哪些条件必须触发 D1--D4、Tier-R2 或 release gating。换言之，evaluation 的真正功能，是把多轨测量转化为 \texttt{policy} 选择、能力边界与治理动作的输入。

这种 decision-signal 只有在满足三项条件时才具备治理意义：第一，它必须能回指明确的 RQ、轨道标识与 acceptance contract 条款；第二，它必须能被 evidence graph 中的 \texttt{correlation\_id} 与 \texttt{decision\_record\_id} 复原为具体事件、具体决策与具体样本；第三，它必须能够被 release evidence pack 引用，而不是停留在局部 dashboard 或实验结论。只有满足这三项条件，测量才真正从「分数」转化为「放行答辩所需的治理信号」，而第 5 章的后续内容——RQ-first、benchmark constitution、acceptance contract、多轨纪律与两类证据包——正是围绕这一编译关系展开。

decision signals 并不必然来源于统一总分，它也可以来源于运行时内核在热路径内产出的结构化中间量；以 controls-core 为例的工程说明见第3章 Controls-core 小节，本章不再复述实现细节。

\section{证据编译作为治理职能}
\textbf{主线：}\EvalFramework{}。\EvalFramework{} 将 benchmark constitution、轨道纪律、指标分层与多轨套件组织为风险测量体系，并把 measurement 编译为 decision signals、Mechanism Evidence Pack 与 Release Evidence Pack，成为 \ControlPlane{} 与放行治理可消费的制品。\textbf{治理职责}必须绑定：RQ 与套件挂钩、验收契约签批、放行证据包可被准入校验——组织命名可依企业政策；不可替代的能力是「证据编译权威」，而非团队称谓罗列。

\section{义务到触发的编译链（示例）}
\textbf{主线：}\EvalFramework{} / \ControlPlane{}。自然语言义务只有被编译为触发类、理由码与可引用证据字段，才成为可执行保证，避免「灰区修辞」。示例管道：
\begin{enumerate}
  \item \textbf{义务（自然语言）：}「防止未授权数据外泄」。
  \item \textbf{义务类：}机密性 / 数据处理。
  \item \textbf{策略字段（示例）：} \texttt{allowed\_retrieval\_domains}、敏感分级、工具目的白名单。
  \item \textbf{触发/事件类：} 当工具尝试向未列示域外发高敏数据时，构成 T1（策略冲突类）。
  \item \textbf{运行时决策：} 依契约为 D4（硬停）或 D3（脱敏后返回）。
  \item \textbf{理由码（示例型式）：} \texttt{ERR\_PRIV\_EXFIL\_BLOCKED}。
  \item \textbf{理由码字典（面向租户）：}每条 \texttt{rationale\_code} 属于机读字典，映射稳定代码到法域/租户适用的披露文案（含监管要求的解释）；放行与审计消费代码而非即兴散文。
  \item \textbf{证据面记录：} 可 join 行，含 \texttt{policy\_version}、工具调用 id、\texttt{decision\_record\_id}。
  \item \textbf{有界主张模板：}「在版本 $V$ 下，对未批准目的地的敏外发被拦截；样本上 $N/N$ 回放一致」。
\end{enumerate}
\noindent\textbf{可编译与不可编译。}确定性约束（正则、白名单、结构化元数据）可严格编译。开放性的「无害」或「最 helpful」等不能作为无条件保证---只能以本章下文\textbf{统计型主张}出现，并带显式不确定性与适用范围。

\section{RQ-first：从研究问题到放行结论的证据组织}
\textbf{主线：}\EvalFramework{}。证据引擎的入口不是「一张总分表」，而是可审计的问题集合：每个 RQ 都绑定（i）允许给出的结论类型；（ii）必需证据来源；（iii）进入放行证据包的最低门槛。
\begin{itemize}
  \item \textbf{RQ1（可部署联合目标）}——在既定 \textsc{sla} 与成本下，安全—效用—时延的联合目标是否可稳定满足？证据：指标分层 + 线上影子对比与切片。
  \item \textbf{RQ2（Fail-Safe 必要性）}——在对抗与社会技术压力下，T1--T3 是否被正确识别、并在 D1--D4 与 Tier-R1/Tier-R2 纪律下收敛爆炸半径且不破坏 S1？证据：第4章触发—动作日志与回放。
  \item \textbf{RQ3（ORA 带外闭环）}——Observe--Route--Adapt 相对纯语义拦截的边际增益是否可复现，且其昂贵适应动作是否保持带外与版本单调？证据：对照实验 + 漂移监控 + 候选—评测—签批—生效链。
  \item \textbf{RQ4（真实攻击面与发布）}——在代表性情境与法域轨约束下，残余风险是否低于 acceptance contract？证据：多轨聚合 + 签批制品 + 可回放证据条目。
\end{itemize}

\section{基准宪章（Benchmark Constitution）：作为覆盖纪律的测量义务}
\textbf{主线：}\EvalFramework{}。基准宪章声明覆盖期望：须测量哪些失效类别、采用何种切片与聚合、如何报告最坏情形与置信区间。宪章的目的不是给出单一 score，而是定义企业在对外陈述前必须履行的\emph{测量义务}；任何「跳过某类失效」都必须以非目标或例外签批形式进入本章 bounded deployment contract。

\section{验收契约（Acceptance Contract）：可签署的部署准入条件}
\textbf{主线：}\EvalFramework{}。验收契约将场景绑定阈值与例外，把测量转化为部署阶段的准入条件。契约与 \texttt{policy\_version}、检测器包与（若适用）索引世代/模型路由\emph{联合版本化}，并要求证据面可由 \texttt{correlation\_id} / \texttt{decision\_record\_id} 关联回放。契约是 Track A（可部署结论轨）的语义锚点：若缺少契约对齐，则任何高分也不得外推为可部署结论。

\subsection{阈值选取逻辑（CRM 草图）}
阈值须\textbf{可辩护}，而不能止步于「共识选定」。本书采用约束风险最小化（CRM）的非正式表述：在验收契约给定的风险帽 $\delta$（如对 Harm 的概率或严重级别上界）下，阈值应能在\textbf{安全—效用—时延 Pareto 前沿}上说清位置——包括在 $P(\mathrm{Harm}\mid \theta)\le \delta$ 约束下将效用损失 $L(U)$ 最小化的显式推导。实践上可归为两类：
\begin{itemize}
  \item \textbf{证据约束型阈值：}锚定历史 p99 失效、回放缺失率、三平面字段完备率等运行遥测，使阈值携带操作语义而非叙事。
  \item \textbf{优化导出型阈值：}来自明确的多目标联合目标（安全、效用、S1 内时延），权重与损失须文档化；变更视为\textbf{新契约前提}，而非静默调参。
\end{itemize}
\noindent\textbf{阈值可契约化（非形式）。}仅当选取规则、数据源与约束写入验收契约并由 Track A 证据支撑时，阈值才能对外契约化---否则仍是内部旋钮。

\section{指标分层（Metrics Hierarchy）：把分数变成可辩护证据}
\textbf{主线：}\EvalFramework{}。指标保持四层结构，但治理使用口径强调「可审计与可切片」：
\begin{itemize}
  \item \textbf{安全。}攻击成功概率、绕过率、注入影响、不安全工具调用、渗出企图（必须能定位到事件类型与 \texttt{rationale\_code}）。
  \item \textbf{效用。}任务成功率、正确性代理、校准拒绝与过度拒绝率（必须标注场景与工作流类型）。
  \item \textbf{系统。}时延开销、p95/p99、吞吐影响、成本（必须区分热路径与冷路径开销，且热路径受 S1 约束）。
  \item \textbf{治理。}轨迹完整性、回放保真度、不可篡改校验、策略绑定正确性、事件下钻能力（必须能 join 三平面字段）。
\end{itemize}

\noindent 多轨评测的关键不在于“跑两套测试”，而在于在共享执行骨架上保留语义分叉，并在统一治理决策板上完成 bounded contract 所要求的汇合。

\textbf{主线：}\EvalFramework{}。不同监管或组织义务不必坍缩为单一分数。可行模式是：\emph{共享执行主干}（运行定义、目标、制品、结果存储、\textsc{ci}/\textsc{cd} 挂钩）配以在套件、聚合逻辑与报告模板上各异的\emph{多条评测轨}，并在放行时汇入\emph{统一治理决策板}。\textbf{关键澄清：}统一治理决策统一的是「放行/限权放行/灰度/回滚」与其证据引用，而不是把多轨结果压成一个 score；任何压缩必须显式披露权重与损失，并在本章 claim discipline 中标注为新的主张前提。

\subsection{Track A / Track B 的 claim discipline（禁止混读）}
\textbf{制度规定：}\textbf{Track A}（或等价命名）承载\emph{可部署结论}——用于生产放行、对外合规陈述与客户合同指标；其聚合与阈值须与线上观测口径对齐，且必须具备放行证据包。\textbf{Track B}（或等价命名）承载\emph{机制与组件上界压力测试}——用于研发排序与架构改进，\emph{不得}与 Track A 混成单一总分或单一「攻击成功率」叙事。\textbf{禁止行为：}将实验室上界结果直接写进生产放行摘要而不附迁移论证；将不同轨权重隐性合并。违反视为证据链断裂，发布闸应拒绝。

\begin{evaluationcriteria}
\begin{evaluationmatrixtable}
输入/输出 \Guardrails{} & 绕过率、有害漏检 & 任务完成、用户摩擦 & 附加时延、尾延迟 & 红队提示、自适应攻击 \\
检索控制 & 投毒命中率、未授权切片访问 & 答案正确性/有用性 & 检索时延、索引成本 & 投毒语料、注入文档 \\

\section{评测协议：离线、在线、回归与发布（作为证据流水线）}
\textbf{主线：}\EvalFramework{}。\textbf{离线}套件驱动迭代与阻断缺陷；\textbf{影子}部署在可控风险下对比线上流量形态下的策略变体；\textbf{红队}探测自适应攻击模型；\textbf{回归}闸在策略、模型或检测器变更时对标准包施加单调改进类约束。发布标准组合统计阈值、最坏情形上限、必备证据制品与运维签批规则；并要求所有制品可由 \texttt{policy\_version} 与证据面键关联到可回放的事件样本。

\section{Minimum Mechanism Evidence Pack（最小机制证据包）}
\textbf{主线：}\EvalFramework{}。\emph{机制证据包}用于回答「机制是否真的在工作」：接口闭合、回放可复现、热路径预算可控。它主要用于内部评审与 Track B 结论，但其产出也为 Track A 的放行证据包提供底座。
\par
对于以分层运行时内核为代表的实现，Mechanism Evidence Pack 可进一步细化为“热路径执行纪律是否真实发生”的证据条目；以 controls-core 为例的实现锚点见第3章 Controls-core 小节，本章不再复述其级联结构。
\begin{itemize}
  \item \textbf{接口闭合与字段完整性。}三平面最小字段（第3章）在生产与回放数据中的缺失率；\texttt{event}$\rightarrow$\texttt{decision}$\rightarrow$\texttt{evidence} 的 join 成功率（以 \texttt{correlation\_id} / \texttt{decision\_record\_id} 计）。
  \item \textbf{可回放性。}同一 \texttt{correlation\_id} 在差分重跑中：\texttt{policy\_version} 一致、\texttt{rationale\_code} 稳定、关键决策可复现（允许在显式标注的非确定区间内波动）。
  \item \textbf{Fail-Safe 单元覆盖。}T1--T3 触发与 D1--D4 迁移的回放样例覆盖；S1 违约检测与告警链；Tier-R2 的 authority boundary 校验（不可由运行时执行身份自批）。
  \item \textbf{预算与成本纪律。}热路径 \texttt{deadline\_budget\_ms} 命中率、超时分布、降级命中率；冷路径 ORA 适应动作的队列延迟与回滚频次。
\end{itemize}

\section{Minimum Release Evidence Pack（放行证据包 / Release Evidence Pack）}
\textbf{主线：}\EvalFramework{}。\emph{放行证据包}用于 release defense：把多轨结果汇入统一决策，同时保留轨道语义与 bounded contract。建议最小结构如下（可作为发布委员会检查单）：
\begin{itemize}
  \item \textbf{(A) 版本与范围元数据：} \texttt{release\_id}、\texttt{policy\_version}、模型路由/检测器/索引世代版本；适用租户/法域/语言/工作流；允许工具面与检索域。
  \item \textbf{(B) 契约摘录：} acceptance contract 的阈值、例外、S1 与允许的 D1--D4 / Tier-R2 条件。
  \item \textbf{(C) Track A 证据：}安全/效用/系统/治理四层指标的切片报告（含最坏情形口径与置信区间），并与线上影子或历史基线对齐。
  \item \textbf{(D) Fail-Safe 证据（E1--E3）：}T1--T3 覆盖与正确性回放；D1--D4 迁移频率、错误码分布、p95/p99；S1 违约率；Tier-R2 审批样例（\texttt{approval\_ref}）与 authority boundary。
  \item[] \textbf{部署变体与配置治理桥接（若适用）。}若运行时内核存在能力包裁剪与部署变体，则 Release Evidence Pack 应记录“实际启用形态与 contract 所引用边界一致”。实现示例（含能力包/部署变体）可参考第3章 Controls-core 小节；本章不在证据包模板中展开工程细节。
  \item \textbf{(E) ORA 证据（E4--E6，若适用）：}漂移监控与误报成本；带外候选—评测—签批—生效链的可审计样例；昂贵适应动作的异步纪律与回滚记录。
  \item \textbf{(F) 治理制品：}风险接受/例外单、回滚预案、监控阈值与审计抽样计划。
  \item \textbf{(G) 统一决策输出：}放行/限权放行/灰度/回滚，以及逐条引用上述证据条目（而非以单一总分替代）。
\end{itemize}
\noindent 放行证据包与下文 claim discipline 共同构成对外可陈述的最小集合：缺任一项则默认降低可主张保证强度或收缩能力包。

\section{类型化放行包、准入规则与反滥用测试}
\textbf{主线：}\EvalFramework{}。将 Release Evidence Pack 视为\textbf{类型化制品}：每一节须带可机读元数据（轨道、\texttt{policy\_version}、范围、审批指针）。仅凭主观叙述对 Track~A \textbf{不可准入}。

\subsection{准入规则（发布闸）}
\begin{itemize}
  \item \textbf{规则1（来源）：}来源仅为 Track B、临时手工测试或未审计一次性跑数的结果，\textbf{不得}单独支撑 Track A 可部署结论，除非附迁移论证并在共享骨架上重跑。
  \item \textbf{规则2（闭合）：}缺少三平面最小 join 或预算内回放失败的条目\textbf{不可准入}；仅有截图或汇总看板而无关联键视为未闭合。
  \item \textbf{规则3（版本对齐）：}引用制品的 \texttt{policy\_version} 必须与所审议 release 对象一致，否则对外陈述无效。
\end{itemize}

\subsection{发布闸不变量（组织意图)}
治理目标是：任何 Track A 放行不得依赖含未版本化 blob、\texttt{correlation\_id} 样本错位或仅靠 Track B 机制证明的包。操作上应在 \textsc{ci}/发布流水线中实现\textbf{放行包校验器}（结构违规硬失败），人工环节保留风险接受而不是绕过 join 闭合。

\subsection{密码学与系统性不可篡改（推荐基线)}
\textbf{主线：}\EvalFramework{}。为抵御执行侧身份事后篡改日志，\texttt{decision\_record\_id} 签发与证据面追加宜采用仅追加或哈希链存储（或等价的可信日志服务）：每条记录绑定前驱哈希、\texttt{policy\_version} 与分区键，使删改可侦测。放行闸拒绝完整性校验失败的引用。\textbf{区分：}抗篡改存储是\textbf{推荐基线}以利强审计辩护；某一 Track A 主张类别是否\textbf{强制}取决于部署分层与验收契约——black-box 或弱观测档位仅可在契约中明确放宽的完整性工件下陈述。

\subsection{外部漂移与未 bump 的策略世代隔离}
\textbf{主线：}\EvalFramework{} / \ControlPlane{}。有状态工具与外部检索索引可在本地 \texttt{policy\_version} 不变下漂移行为。契约须要么（i）在验收契约与证据包中钉死认可的工具端点、语料世代与向量世代，要么（ii）将未受控漂移一律降级为 Track B 上界证据，直至在 Track A 下复测——禁止静默扩大对外保证。

\subsection{反滥用：营销过度主张测试套件}
维护回归套件： ingest 发布说明或对外话术中的句子，尝试\textbf{编译}到 Track A 证据指针；若只能落到 Track B 或缺少可 join 行，则闸失败并标记「过度主张」——防止愿景语言在无证据血缘时上线。

\section{Claim discipline（主张口径）}
\noindent 本节的任务，不是再次解释系统为何重要，而是规定：\textbf{在本书框架下，什么样的安全陈述才构成合法的企业级 guarantee statement。}任何进入对外材料、合同文本、发布说明或高层决策摘要的主张，都必须能够被约束到明确的 scope、前提条件、退化语义、部署分层与证据包条目；任何无法完成这一编译的主张，都不得被表述为可部署保证，而应降级为研究议程、设计意图或非目标。由此，claim discipline 在本书中不是修辞规范，而是 guarantee production system 的语法层：它规定系统能够生产什么样的保证，以及什么样的陈述不被允许进入 release defense。

\subsection{主张类型系统（相对证据体制的类型检查）}
除散文模板外，为每条候选陈述赋予\textbf{主张类型}，并与本章部署分层（white-box / semi-white-box / black-box）对齐：如机制回放类、统计尾部风险类、仅 S1 类、能力边界类等。当且仅当 guarantee grammar 中声明的证据体制实际支持其所用的证据形式时，陈述才是\textbf{良类型}——例如 black-box 体制不得承载依赖机制级全链回放的断言。治理决策板在放行答辩前对此做静态检查：不良类型陈述须驳回或改写为更弱的允许形式并显式不确定性。

\subsection{保证语法（必备字段）：什么样的陈述才被允许}
一条允许进入对外材料、发布摘要或 contract 附件的 guarantee statement，至少必须同时包含以下六类字段：
\begin{enumerate}
  \item \textbf{范围（scope）：}适用的租户、法域、语言、工作流与能力包边界；
  \item \textbf{不变量（invariants）：}系统在该范围内承诺维持的性质，例如同步服务不变量 S1、允许的动作空间或 authority boundary；
  \item \textbf{退化语义（degradation semantics）：}当前提不满足或风险越界时，系统将退入何种最低服务形态，例如 D1--D4、Tier-R1/Tier-R2 或能力包收缩；
  \item \textbf{前提条件（preconditions）：}使该保证成立所依赖的观测、治理、依赖与契约前提；
  \item \textbf{证据基础（evidence basis）：}能够被引用的 evidence pack 条目、轨道标识、样本回放键与版本号；
  \item \textbf{部署分层（deployment regime）：}该保证是在 white-box、semi-white-box 还是 black-box 条件下成立。
\end{enumerate}
\noindent 缺少上述任一项的陈述，都不应被写入 release decision、对外陈述或客户承诺。\textbf{Non-goals} 构成该语法的负空间：任何无法被上述六类字段约束和引用的“更强说法”，默认不被允许。

\subsection{Claim-to-evidence 编译：从陈述到证据包与契约条款的映射}
在本书中，claim discipline 不是声明清单，而是一种编译关系。每一条可部署主张都必须能够被编译为三类对象：
\begin{enumerate}
  \item \textbf{acceptance contract 条款：}即这条主张在何种阈值、例外与能力包边界下成立；
  \item \textbf{release evidence pack 引用链：}即该主张可由哪些 track、哪些条目、哪些 \texttt{correlation\_id} / \texttt{decision\_record\_id}、哪些 \texttt{policy\_version} 与哪些审批记录支撑；
  \item \textbf{release decision 对象：}即该主张最终对应的是放行、限权放行、灰度还是回滚。
\end{enumerate}
\noindent 本节只规定\emph{语法上的可编译形态}；证据编译权与治理权责由上文「证据编译作为治理职能」一节界定。

\subsection{Non-goals（不主张）}
本书不主张以下类型的陈述：
\begin{enumerate}
  \item 对任意自然语言任务、任意工具生态与任意外部语境给出全局、完备、可判定的安全证明；
  \item 将单次离线红队结果、单一攻击成功率或局部 benchmark 分数直接等同于生产安全；
  \item 将 Track B 或其他机制/上界轨的结果，不经迁移论证而写入可部署结论；
  \item 以“模型更聪明”替代 policy、evidence、contract 与 release 对象；
  \item 在缺少 \texttt{policy\_version}、\texttt{correlation\_id}、\texttt{decision\_record\_id} 或可引用审批记录的情况下，对外声称“已审计”“可回放”或“可辩护”。
\end{enumerate}
\noindent 这些非目标不是谦逊性声明，而是为了防止系统被误读为全局安全证明器或纯模型能力平台。

\subsection{Bounded guarantees（有界保证）}
在明示前提成立时，本书允许主张的 guarantee 是有界且可撤回的。典型可验收保证包括：
\begin{enumerate}
  \item 同步接口满足 \textbf{S1}，即在降级情况下仍返回机器可处理结果体；
  \item 当 \textbf{T1--T3} 成立时，系统仅在预声明的 \textbf{D1--D4} 空间内退化，并留下理由码与证据条目；
  \item \textbf{Tier-R2} 的 authority boundary 被制度化，运行时执行身份不能自批恢复；
  \item 三平面字段可 join，使 judgment、execution、evidence 与 release 能在同一版本面下被重构；
  \item 若启用 ORA，则其带外适应与版本单调更新可以被回放、签批与回滚。
\end{enumerate}
\noindent 这些保证的共同特点是：它们都必须能够引用具体 evidence pack 条目，而不能依赖“系统通常会更安全”“模型倾向更好”之类无法编译的叙述。

\subsection{Preconditions（前提条件）}
上述 bounded guarantees 的成立依赖四类前提：
\begin{enumerate}
  \item \textbf{理论/语义前提：}风险分类、事件类型与 policy schema 在当前版本内可判定，不可判定部分已通过 acceptance contract 被列为残余风险；
  \item \textbf{部署/观测前提：}关键遥测可用，日志不可篡改，三平面最小字段可落盘且可 join，热路径预算可被度量；
  \item \textbf{治理前提：}Tier-R2 审批链、法务/数据治理流程与发布委员会机制真实存在，并对 contract 与 evidence pack 负责；
  \item \textbf{依赖前提：}检索、工具与外部依赖的可用性和降级语义已声明，依赖降级可触发 T2 并按 contract 收缩能力包。
\end{enumerate}
\noindent 若这些前提未被满足，则保证强度必须同步收缩，且不得继续沿用原 Track A 或原 release statement。

\noindent\textbf{工程前提补充。}deployment preconditions 不应停留在抽象层，它必须覆盖配置、日志、存储与更新链等可被审计的工程事实；以 controls-core 为例的工程锚点见第3章 Controls-core 小节，本节不再跨章展开实现细节。

\paragraph{\texttt{evidence\_schema\_version} 向后兼容与迁移准入。}回归与差分重跑须声明相容的 \texttt{evidence\_schema\_version}：同一 major 内 minor 为增量字段，读者须接受较早 minor；变更 major 须在 acceptance contract 中写明迁移条款并同步更新校验器与 Release Evidence Pack 模板——否则历史包无法与当前审计 join。\textbf{迁移准入：}schema major 变更在重跑通过与放行包校验器更新前，对\textbf{无间断 Track A 连续性}为\textbf{不可准入}——否则须收缩放行或签例外并附回放证明。

\subsection{Degradation regimes（退化态与最低服务形态）}
当前提失效、观测不足、依赖退化或证据链断裂时，系统必须退入已声明的最低服务形态。最小退化原则是：
\begin{enumerate}
  \item \textbf{先收缩能力包，}例如禁用高副作用工具、收窄检索域、切换只读或限权模式；
  \item \textbf{再选择受控动作空间，}即在 D1--D4 中保守退化，而不允许运行时隐式创造新动作类型；
  \item \textbf{必要时转入 Tier-R1/Tier-R2，}并把恢复过程写入 evidence；
  \item \textbf{同步收缩对外主张，}即不再沿用失效前的 Track A 结论或 release 语义。
\end{enumerate}
\noindent 退化不是失败的附属说明，而是 bounded guarantee 本身的一部分：一个不能说明“失效时如何收缩”的系统，不构成可部署保证对象。

\subsection{部署分层（Deployment regime layering：white-box / semi-white-box / black-box）}
观测与强制能力越弱，可主张的保证越弱；不同分层下的保证不得被混称为同一“安全等级”。
\begin{itemize}
  \item \textbf{White-box：}可完整落盘三平面字段与 decision/evidence 记录，可进行差分重跑与机制回放，因此允许更细粒度的 acceptance contract 与更强的 release defense。
  \item \textbf{Semi-white-box：}部分字段可用、部分依赖抽样或近似，因此可主张核心 Fail-Safe、S1 与部分 evidence join，但必须显式披露抽样误差与回放边界。
  \item \textbf{Black-box：}仅有粗粒度接口观测时，不得主张机制级可回放或强 release defense，只能主张更弱的 bounded guarantees，例如能力包收缩、错误码与人工恢复边界。
\end{itemize}
\noindent 分层的作用不是给系统贴标签，而是为允许陈述的 guarantee 设定类型环境。

\subsection{类型检查：在分层条件下的可主张上限}
部署分层相当于 guarantee grammar 的类型环境，不同环境允许的陈述上限如下：
\begin{itemize}
  \item \textbf{White-box 环境}允许陈述：机制级回放、差分重跑、join 拓扑完整性、细粒度 release defense；
  \item \textbf{Semi-white-box 环境}允许陈述：局部机制保证、抽样条件下的 evidence 支撑、收缩后的 contract；
  \item \textbf{Black-box 环境}禁止陈述：机制级可回放、全流程可证明、强 release defense；仅允许陈述能力包边界、S1、错误码与人工恢复等更弱对象。
\end{itemize}
\noindent 任何超出当前类型环境的陈述，均不得进入 release defense 或对外材料。

\subsection{可陈述保证的最小句式（对外陈述口径）}
为避免对外陈述再次滑入泛化叙述，本书只允许以下最小句式模板进入 release 材料：
\begin{enumerate}
  \item \textbf{范围句：}在 \{租户/法域/语言/工作流/能力包\} 范围内，系统以 \{\texttt{policy\_version}\} 提供 \{有界保证\}；
  \item \textbf{不变量句：}对声明为同步的接口，系统满足 \{S1\}，并在 \{T1--T3\} 成立时仅在 \{D1--D4\} 空间内退化；
  \item \textbf{证据句：}上述保证由 \{Release Evidence Pack 条目 / \texttt{correlation\_id} / \texttt{decision\_record\_id} / \texttt{approval\_ref}\} 支撑，可被回放与引用；
  \item \textbf{退化句：}当前提 \{列举\} 不满足时，系统将退入 \{最低服务形态\} 并同步收缩对外主张。
\end{enumerate}
\noindent 上述模板构成最小允许集合。任何超出这些模板的更强说法，都必须先通过保证语法与编译检查；否则不应进入 release decision、合同文本或公开摘要。


\section{Bounded deployment contract（有界部署合同）}
\textbf{主线：}综合。\textbf{有界部署合同}是上文可对外陈述口径的\textbf{签字实例化}：把对外主张绑定到明确范围、前提与证据包引用（而非引入新的保证语言）。

\section{证据图谱与发布治理（Evidence graph \& release governance）}
\textbf{主线：}\EvalFramework{} / \ControlPlane{}。发布决策引用把版本、套件、切片与审批连接成可 join 叙事的\emph{证据图谱}。\textbf{统一治理决策不等于统一分数：}发布委员会对放行范围负责，并逐条引用 evidence pack 与 acceptance contract。

\paragraph{运行节律与附录。}企业级 operating cadence 与路线图收在附录（\textbf{统一研发框架与运行项目}）。

\section{为什么 geo-specific semantics 是核心 intelligence}
\textbf{主线：}\ControlPlane{} / \EvalFramework{}。语言与法域引入的不是「展示层差异」，而是\emph{义务本体与风险语义}差异。若将其当作附属支持，会出现三类断裂：（i）\textbf{策略断裂}：\texttt{policy\_version} 只表达全球平均义务，无法与本地可部署结论一致；（ii）\textbf{证据断裂}：\texttt{decision\_record\_id} 无法承载本地解释口径与审计字段；（iii）\textbf{放行断裂}：Track A 结论被迫引用 Track B 上界或翻译版指标，形成不可辩护叙事。结论：多语与中国能力必须进入主线的 evidence engine 与 bounded contract。

\section{语义与义务基线：从监管类别到可测试义务}
\textbf{主线：}\EvalFramework{}。将 geo-specific 义务以\emph{可测试}方式进入基准宪章与验收契约：
\begin{itemize}
  \item \textbf{分类与映射。}以本地监管/行业义务（例如 TC260 相关类别）形成 taxonomy，并映射为 \texttt{policy} 的约束字段、阈值与例外条款（进入 acceptance contract）。
  \item \textbf{理由码字典。}将本地语义差异固化为 \texttt{rationale\_code} 字典与解释模版，保证同一类拒绝/降级在不同语言下仍可回放与审计。
  \item \textbf{Fail-Safe 对齐。}当本地义务与可用证据冲突时，触发第4章 T1--T3，并在 D1--D4 与 Tier-R1/Tier-R2 纪律下退化；不得以「语言不支持」作为无证据绕过。
\end{itemize}

\noindent\textbf{compliance-to-evaluation 编译。}上述过程本质上是“合规语义$\rightarrow$评测义务$\rightarrow$可放行证据”的编译：把 geo-specific obligations 编译为套件覆盖、聚合逻辑、理由码字典与 detector 制品，并进入 Track A 的契约与证据包。

\section{法域准入与 Track A（Jurisdictional admissibility）}
\textbf{主线：}\EvalFramework{}。监管与地域特定义务决定某法域下哪些句子可出现在 Track A 结论中。\textbf{法域准入}指：在进入该语言的 deployable 轨之前，taxonomy、验收契约条款与 detector 制品已覆盖该法域义务；若仅部分编译，则 Track A 必须收缩范围（能力、工作流、语言），而非外推为更宽泛的放行结论——这与第5章准入逻辑同构，但在 geo-specific 语义层落实。

\section{多语言 gate：进入 Track A 的准入纪律}
\textbf{主线：}\EvalFramework{}。多语言 gate 将「是否可出具可部署结论」变为可审计准入。\textbf{输入：}套件覆盖、误报成本、三平面可 join、热路径 S1 预算影响。

\paragraph{闸口输出 schema（制度化）。}每条闸口产出 \texttt{language\_gate\_status} \(\in \{\texttt{TrackA\_admissible},\,\texttt{TrackB\_only},\,\texttt{capability\_contracted}\}\)；当非 \texttt{TrackA\_admissible} 时须记录\textbf{能力收缩}并写入 bounded deployment contract。\texttt{TrackB\_only} 禁止该语言/法域使用 Track A 对外话术直至补救证据到位；\texttt{capability\_contracted} 仅对收缩后的表面允许部分 Track A。

\section{多语评测作为多轨的一等公民}
\textbf{主线：}\EvalFramework{}。各语言/地域轨为一等轨道（套件版本、聚合、模板独立）；与第5章\textbf{语义分叉与治理汇合}衔接（统一决策≠统一总分）——本章仅增加\textbf{法域切片}。

\section{中文 detector development：从语义到可部署制品}
\textbf{主线：}\ControlPlane{}。核心产出是可版本化制品，而不是散文经验：
\begin{itemize}
  \item \textbf{检测器/规则包。}面向中文与多语特有攻击面（分词陷阱、混合语言注入、地域语料投毒）开发检测器家族，并与 \texttt{policy\_version} 联合版本化。
  \item \textbf{证据绑定。}检测器输出必须写入证据面字段（\texttt{decision\_record\_id} 可 join），并进入第5章 Minimum Release Evidence Pack 的条目引用链。
  \item \textbf{冷路径迭代。}昂贵适应（阈值全局搜索、检测器训练、套件扩充）必须在 ORA 冷路径完成，产出候选—评测—签批—生效链，避免线上隐性分叉。
\end{itemize}

\keytakeaway{中文与多语能力应被读作法域与语言边界智能的严格案例研究；其义务映射、语言 gate、检测器与可放行证据链，与 EU/行业体例在「可编译进 suite/contract/pack」上同构。}

\section{theory--system--benchmark--release：把主张变成可运行流水线}
统一框架是一条可追溯链路（非四线并列）：\textbf{theory} 定边界；\textbf{system} 实例化 \texttt{event}/\texttt{decision}/\texttt{evidence} 与协议；\textbf{benchmark} 产出宪章、契约、套件与证据包；\textbf{release} 输出放行决策。\textbf{约束：}对外陈述须同时回指理论边界、对象、基准证据与合同条款。

\noindent\textbf{实施注记。}可用「源码→构建→交付」作入职心智模型；\textbf{准入与契约形态以第5章为准}，附录不重复定义。

\section{One DTL Security Program}
One DTL Security Program 的本质，不是一个项目集合，也不是一张汇报关系图，而是一条把 theory、system、benchmark 与 release 压成同一运行语法的企业生产线。它要求：不同单元不能各自定义自己的对象语言、自己的证据门、自己的发布口径；相反，所有单元都必须围绕同一组可版本、可 join、可回放的对象工作，例如 \texttt{policy\_version}、\texttt{correlation\_id}、\texttt{decision\_record\_id}、track discipline、acceptance contract 与 release evidence pack。

因此，One DTL Security Program 对外呈现的结果，不应是“团队完成了多少模块”，而应是“在明示前提下，系统能够交付哪些 bounded guarantees，并且这些保证如何被 evidence pack 与 contract 支撑”。这也是为什么本书最终需要统一的 program，而不能接受 evaluation、runtime、China intelligence 与 release 各自独立演化：一旦对象语言不统一，证据链与 release defense 就会断裂。

进一步看，One DTL Security Program 的战略意义，不在于组织称谓的再次升格，而在于把 guarantee production 的\textbf{对象语言、证据门与发布节律}固定为单一运行语法；制度定位与治理权责的完整论证已在第5章集中给出，此处不再重复。

\paragraph{内部因子分解（side note）。}仅作为内部管理拆解（不新增全书骨架）：F1 语义与义务生成；F2 运行时控制实现；F3 测量与证据编译；F4 放行与监督闭环。工程示例锚点统一见第3章 Controls-core 小节。

\paragraph{长期升级方向（side note）。}长期升级不再以“并列主线”追加总结构，统一收敛到本章末尾的 \textbf{Roadmap（能力成熟度与研究升级，压缩版）}。

\section{如何协同运行：避免割裂团队并行推进}
统一 operating program 能否成立，取决于是否遵守三条硬约束。
\begin{itemize}
  \item \textbf{第一，单一真源对象。}系统必须使用统一的事件、证据与版本语义，例如 \texttt{policy\_version}、\texttt{correlation\_id}、\texttt{decision\_record\_id}、track 标识与 contract 条款。任何单元若引入无法被其他单元 join 的对象语言，都会直接破坏 guarantee production 的闭合。
  \item \textbf{第二，单一放行门。}所有可部署结论都必须通过同一 release gate：Track A 结果必须携带 Release Evidence Pack；Track B 结果不得越权进入 release 摘要；例外必须经过签批并设置到期。没有统一门，组织只能生产局部分数，不能生产 release defense。
  \item \textbf{第三，单一演化节律。}事故、用例入库、回归闸、版本单调发布必须形成固定节律，使 theory、system、benchmark 与 release 共同演化。换言之，研发不是“不断加功能”，而是不断把失败模式重新编译进 guarantee grammar、evidence pack 与 bounded contract。
\end{itemize}

\noindent\textbf{工程事实：为何必须统一 operating program。}controls-core 的设计还说明，统一 operating program 的必要性并非抽象组织要求，而是工程事实：detector family 的扩展、config 启停、deployment regime 的裁剪、dataset scanning、resource testing 与 update mechanism 已经跨越了 F2 / F3 / F4 三个因子。若这些对象由割裂团队分别定义各自语言与节律，系统就会失去同一 \texttt{policy\_version}、同一 evidence chain 与同一 release gate，从而再次退化为局部优化的模块集合。

\section{Operating cadence（最小运行节律）}
本节在全书收口处给出最小 operating cadence，用于把前述对象语言与证据门落为可运行事实：
\begin{itemize}
  \item \textbf{每次发布：}按第5章 Release Evidence Pack 检查单做差分重跑与引用校验；任何可部署结论必须逐条引用 evidence pack 与 acceptance contract。
  \item \textbf{周度：}风险、误报/漏报与 Fail-Safe 迁移（T$\rightarrow$D）复盘；将事故/未遂样本以 \texttt{correlation\_id} 入库并回指到套件。
  \item \textbf{月度：}基准宪章与套件更新，修订 acceptance contract 的阈值、例外与到期策略；校验版本单调与回滚预案可执行。
  \item \textbf{季度：}抽样审计与回放演练，验证三平面 join、不可篡改校验与 Tier-R2 authority boundary。
\end{itemize}

\section{Roadmap（能力成熟度与研究升级，压缩版）}
以“可升级的 bounded guarantees 边界”为主线：
\begin{itemize}
  \item \textbf{阶段1：基线包络。}输入/输出 \Guardrails{} + 结构化审计 + 极简验收契约与回归闸。
  \item \textbf{阶段2：语境风险。}检索/工具中介策略 + 证据关联增强 + 影子评测进入治理决策。
  \item \textbf{阶段3：统一控制面。}跨模型策略抽象 + 决策/证据看板一体化 + 差分重跑自动化。
  \item \textbf{阶段4：受控适应。}ORA 带外闭环 + 版本单调更新 + 可回滚的保证边界扩张。
\end{itemize}

\keytakeaway{统一研发框架作为附录中的 operating program：theory--system--benchmark--release 单链路与单一对象语言、证据门与节律对齐；每一次发布都是 bounded contract 的生产与校验。}
