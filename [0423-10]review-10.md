Chapter 8. 统一研发框架：theory–system–benchmark–release operating program

最终压缩版可替换稿（建议版）

**Chapter Objective**本章的任务，不是再介绍组织结构，而是把全书收口为一个可运行的 operating program。前文已经依次定义了总对象（Unified AI Security Control Plane）、系统原理（Decoupled Safety）、证据引擎（Evaluation Framework）、bounded deployment contract 与 geo-specific semantics；本章要回答的是：这些对象如何不再停留于章节结构，而被压成同一条可持续运行的企业生产线。也就是说，本章讨论的不是“谁负责哪个模块”，而是**理论如何变成可主张性质，性质如何变成系统对象，系统对象如何变成 benchmark 与 evidence，evidence 又如何变成 release decision 与 bounded contract**。One DTL Security Program 的意义，正是在于用统一对象语言、统一证据门与统一节律，把这条链路从概念蓝图变成组织可运行事实。

**Deliverables**

* 以 theory → system → benchmark → release 的单链路，收口全书对象，而不是再列并行条目。
* 明确 One DTL Security Program 的最小运行约束：单一真源对象、单一放行门、单一演化节律。
* 将统一研发框架表达为保证生产的因式分解：语义与义务生成、运行时控制实现、测量与证据编译、放行与监督闭环。

8.1 theory–system–benchmark–release：把主张变成可运行流水线

本书中的统一研发框架，不是“研究、工程、评测、发布”四项工作的并列排布，而是一条单向可追溯、双向可回流的生产链。**theory** 负责给出 claim discipline、guarantee grammar、geo-specific semantics 与可主张边界；**system** 负责把这些边界压到 event、decision、evidence 与 T/D/S/Tier-R/ORA 等可验收对象；**benchmark** 负责把这些对象转译为 benchmark constitution、acceptance contract、多轨套件与两类 evidence pack；**release** 则负责把证据输入统一治理决策板，并输出放行、限权放行、灰度、回滚等 bounded deployment contract。

这条链路的关键约束是：**任何可对外陈述的结论，都必须能够同时回指到 theory 中定义的主张边界、system 中实现的对象、benchmark 中产生的证据，以及 release 中被引用的 contract 条款。** 一旦这条链路中断，系统仍然可以“做很多安全能力”，但不能持续生产可辩护 guarantee。由此，本章并不把研发框架理解为资源编排，而是把它理解为 guarantee production system 的组织形式。8.2 One DTL Security Program

One DTL Security Program 的本质，不是一个项目集合，也不是一张汇报关系图，而是一条把 theory、system、benchmark 与 release 压成同一运行语法的企业生产线。它要求：不同单元不能各自定义自己的对象语言、自己的证据门、自己的发布口径；相反，所有单元都必须围绕同一组可版本、可 join、可回放的对象工作，例如 policy_version、correlation_id、decision_record_id、track discipline、acceptance contract 与 release evidence pack。

因此，One DTL Security Program 对外呈现的结果，不应是“团队完成了多少模块”，而应是“在明示前提下，系统能够交付哪些 bounded guarantees，并且这些保证如何被 evidence pack 与 contract 支撑”。这也是为什么本书最终需要统一的 program，而不能接受 evaluation、runtime、China intelligence 与 release 各自独立演化：一旦对象语言不统一，证据链与 release defense 就会断裂。8.3 四个一级因子：职责与核心 artifact

从保证生产的角度看，统一研发框架可以被压缩为四个不可约因子，而不是四个并行团队。

**因子 F1：Theory & Semantics（语义与义务生成）**这一因子的职责是定义可主张的边界：风险分类、义务语义、Non-goals、Preconditions、Degradation regimes、geo-specific semantics 与 rationale_code 体系。其核心 artifact 不是“研究结论”，而是 taxonomy、obligation mapping、guarantee grammar 与可引用的 claim 模板。

**因子 F2：System & Kernel（运行时控制实现）**这一因子的职责是把理论对象压到系统对象：DECOUPLED SAFETY KERNEL 最小接口、事件类型、强制挂点、T/D/S/Tier-R/ORA 的实现与回放 harness。其核心 artifact 是 kernel API、策略 DSL、执行挂点规范、回放工具与差分重跑链路。

**因子 F3：Benchmark & Evidence（测量与证据编译）**这一因子的职责是把可主张边界与系统对象编译成 benchmark constitution、acceptance contract、多轨套件、Mechanism Evidence Pack 与 Release Evidence Pack。它的核心 artifact 不是“评测报告”，而是 RQ-first 映射表、套件版本、轨道纪律与 evidence pack 模板。

**因子 F4：Release & Operations（放行与监督闭环）**这一因子的职责是把证据变成治理行为：统一治理决策板、差分重跑策略、例外到期机制、SRE / IR 闭环与审计抽样。它的核心 artifact 是 release checklist、决策记录、回滚预案、审计计划与 Tier-R2 authority boundary 配置。

这四个因子共同构成的不是“组织图”，而是企业 AI 安全保证生产的因式分解：它们分别生成语义、实现控制、编译证据、执行放行，但必须共享同一对象语言与同一 release gate。8.4 三条长期研究主线：以“可升级的保证边界”组织研究

如果前述四个因子是生产结构，那么长期研究主线就是这套生产结构的升级方向。本书建议将长期研究组织为三条围绕“可升级 guarantee 边界”的主线，而不是围绕孤立技术主题。

**主线一：可主张保证的扩大。**目标是把 black-box / semi-white-box / white-box 不同 deployment regime 下可陈述的 bounded guarantees 逐步扩大：更多对象可回放、更多决策可被证据化、更多 release statement 可以被精确 contract 化。

**主线二：非确定与多目标权衡的证据化。**目标是把安全、效用、时延、成本与审计性之间的张力从经验折中，提升为可被 evidence pack、置信区间与最坏情形口径共同描述的治理对象。换言之，不是只优化系统，而是优化“哪些 trade-off 可被合法主张”。

**主线三：对抗演化与受控适应。**目标是通过 ORA 冷路径、版本单调更新、差分重跑与回滚纪律，证明系统可以在不破坏 release defense 的前提下持续吸收攻击演化，而不是在线产生隐性策略漂移。

这三条主线的共同特征在于：它们都不只是追求更强技术能力，而是追求更强的 bounded guarantees 生产能力。8.5 如何协同运行：避免割裂团队并行推进

统一 operating program 能否成立，取决于是否遵守三条硬约束。

**第一，单一真源对象。**系统必须使用统一的事件、证据与版本语义，例如 policy_version、correlation_id、decision_record_id、track 标识与 contract 条款。任何单元若引入无法被其他单元 join 的对象语言，都会直接破坏 guarantee production 的闭合。

**第二，单一放行门。**所有可部署结论都必须通过同一 release gate：Track A 结果必须携带 Release Evidence Pack；Track B 结果不得越权进入 release 摘要；例外必须经过签批并设置到期。没有统一门，组织只能生产局部分数，不能生产 release defense。

**第三，单一演化节律。**事故、用例入库、回归闸、版本单调发布必须形成固定节律，使 theory、system、benchmark 与 release 共同演化。换言之，研发不是“不断加功能”，而是不断把失败模式重新编译进 guarantee grammar、evidence pack 与 bounded contract。

**Key Takeaway**统一研发框架在本书中不是组织设计说明，而是 guarantee production system 的 operating program。它用 theory–system–benchmark–release 的单链路，把可主张边界、系统对象、多轨证据与放行治理压成同一条生产线；用 One DTL Security Program 确保对象语言、证据门与发布节律不再分裂。由此，系统每一次发布都不只是一次部署行为，而是一次 bounded contract 的生产、验证与升级。
