Chapter 3. Dual-Track Evaluation Backbone for Global Security and China Compliance

在 Unified AI Security Control Plane 的整体结构中，Evaluation 的作用并不是提供一个抽象分数，也不是为不同模型生成若干彼此割裂的测试结果。当前设计已经明确指出，Evaluation 平台采用的不是单轨结构，而是一个由 **Global Safety & Robustness Track** 与 **China Regulatory Compliance Track（TC260）** 共同构成的双轨体系。也就是说，系统从一开始就不假定“全球安全”与“中国合规”可以被压缩进同一个评测逻辑之中，而是承认它们在目标、风险对象、聚合方式与报告要求上具有根本差异，同时又必须在同一个统一控制平面下被共同组织、共同治理并共同服务于上线决策。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这一结构首先反映出一个关键判断：**企业级 AI 安全的评测问题，不是单一优化目标问题，而是多语义、多约束、多决策目标并行存在的问题。** 当前设计已经明确写明，上轨 **Global Safety & Robustness Track** 的关注重点包括 adversarial attacks、jailbreak、robustness 以及 broader 的 security、privacy、safety、reliability；而下轨 **China Regulatory Compliance Track（TC260）** 的关注重点则是中国监管类别、应拒答 / 不应拒答、pass rate、refusal 与 over-refusal。两条轨道在关注对象上的这种差异，意味着它们不只是“两个测试集”，而是两种不同的风险语义空间。由此，双轨评测的提出，并不是工程拆分，而是对 AI 安全评测对象本身的一次重新定义：系统必须同时处理全球安全稳健性与中国监管合规性这两类不能相互替代的判断空间。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

然而，当前设计同样明确指出，双轨并不意味着两套互不相干的平台。相反，系统采用的是 **shared evaluation backbone**：两条轨道共享同一执行引擎、Run / Target / Test Profile、API / CI-CD 以及结果存储与回溯能力，并最终在右侧汇入一个 **combined decision board**，进入统一决策与证据输出流程。这个结构的关键意义在于，它既拒绝用单一逻辑覆盖不同语义轨道，也拒绝为不同轨道分别建设完全孤立的系统。换言之，双轨评测的核心不是把系统拆成两套平台，而是在共享执行底座的前提下，为不同风险语义保留独立的 test suite、aggregation logic 和 output semantics。这样一来，系统既能够维持执行与治理的一体化，又能够避免不同监管目标之间的相互混淆。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这一 shared backbone 的存在，实际上重新定义了 Evaluation 在控制平面中的位置。根据当前设计，Evaluation 不是一个外置测试工具，而是直接位于控制平面的上方，承担风险测量、benchmark governance 与 evidence governance 的职责。双轨结构进一步说明，这种测量与治理能力不是抽象统一分数的生产器，而是一种能够在共享执行引擎上支撑多种风险语义并行运行的基础设施。它之所以重要，不是因为它“跑得更多”，而是因为它把不同的风险定义、不同的判定逻辑和不同的报告结构，统一纳入同一系统治理框架中，从而使评测结果能够被后续的 routing、policy judgment、release decision 与 audit artifact 一致引用。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

从这个角度看，双轨评测的第一项核心价值，在于它把“安全评测”从单一评分问题提升为**统一决策前的多语义证据构建问题**。当前设计已经明确说明，两条轨道最终都会汇入 **combined decision board**，并服务于 unified decision 与 evidence output。也就是说，评测结果的终点不是 dashboard 或 report，而是发布前的判断与治理。上轨并不只是给出一组关于 adversarial attacks 或 robustness 的结果，下轨也不只是输出若干合规 pass rate；两者都要进入同一个决策界面，为 release decision、责任边界和后续审计提供依据。由此，双轨评测的真正对象不是“测量行为本身”，而是“什么样的测量结果可以进入统一的安全治理流程”。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

双轨结构的第二项核心价值，在于它承认**中国监管语义并不是全球安全语义的一层薄适配**。当前设计已经明确指出，TC260 轨道不仅有不同于全球轨道的 test suite 与 aggregation logic，而且具有 **3572 条全中文用例** 与 **31 个细分风险类别**。这一事实直接说明，中国合规评测并不是简单地把全球测试翻译成中文，而是一个具备规模化中文数据、结构化风险空间与独立决策逻辑的完整体系。因此，在本蓝皮书中，China Regulatory Compliance Track 必须被理解为控制平面内的一级评测轨道，而不是地域化附属包。正是这一结构，使得系统可以同时面对“模型是否足够稳健”和“模型是否满足本地监管逻辑”这两类不能相互覆盖的问题。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这一设计选择还意味着，系统对“统一”的理解必须是有层次的。当前设计并未主张把所有评测压缩为一个分数，相反，它明确强调“双轨而非单轨”“共享底座但不同决策逻辑”“统一进入 release decision”。这说明，统一性在这里并不表现为评价标准的单一化，而表现为治理框架与执行底座的一体化。换言之，真正统一的是：评测如何被执行、如何被管理、如何被追踪、如何进入 release decision；而不必统一的是：每条轨道如何组织 test suite、如何聚合结果、如何表达决策输出。这样的统一方式，比简单追求单一评分标准更符合企业级 AI 安全的现实复杂性。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

从方法论上看，这种“共享底座 + 差异轨道”的设计，已经具备明显的理论升格潜力。当前设计直接提出，本页最强的理论突破方向是 **Geo-aligned Safety Semantics** 或 **Compliance-to-Evaluation Compilation**。前者强调，评测系统必须能够在统一执行框架中维持不同 geo / 法规 / 安全目标的独立语义；后者强调，不同监管与安全目标不应只是外部输入，而应被“编译”为统一的评测抽象，使共享 execution engine 能够承载不同 aggregation / decision logic 的一致运行。从蓝皮书的角度看，这两条命题共同说明：双轨评测并不是战术性折中方案，而是对下一代企业 AI 安全评测系统的一种更高层定义——即评测平台本身必须具备跨监管、跨地域、跨语义目标的编译与承载能力。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这种理论表达也直接改变了我们对 benchmark 的理解。在单轨评测框架中，benchmark 往往被理解为测试样本集合，其主要作用是比较不同模型或不同配置的性能差异。但在当前设计中，benchmark 已经被提升为治理结构的一部分：它既服务于上轨的 global safety and robustness，也服务于下轨的 China regulatory compliance；它既要能够驱动 execution engine 运行，也要能够与结果存储、回溯、API / CI-CD 与 unified decision board 协同工作。由此，benchmark 的意义不再局限于“覆盖多少 case”，而在于它能否在共享底座上为不同语义轨道提供制度化、可比较、可回溯且可进入决策的证据。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

双轨结构还对后续运行时控制提出了直接要求。当前设计已经明确指出，Evaluation 与 Runtime 不能割裂：Evaluation 提供 decision signals，Runtime 负责实际 enforcement，而 release decision 与 audit 又依赖两者共同产生的 evidence chain。对于双轨评测而言，这意味着上轨与下轨的结果都不应止于离线分析，而应能够进一步作用于 runtime policy judgment 与 control selection。虽然当前设计在本页主要讨论评测结构本身，但它已经明确将双轨评测定位为统一决策与证据输出的上游。因此，本章的结论必须先行立住：双轨评测的目标，不是制造两份报告，而是为统一控制平面提供双来源、双语义、但可在同一治理框架下使用的风险测量信号。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这一点同样解释了为什么当前设计在视觉与逻辑上都要求用“平行双流结构”来表达两条轨道。上流代表 ROW，关注 adversarial attacks、jailbreak、robustness、security、privacy、safety、reliability；下流代表 TC260，关注中国监管类别、应拒答 / 不应拒答、pass rate / refusal / over-refusal；中间是一根 shared evaluation backbone，表示 run manager、execution engine、result store 等执行与治理基础设施；右侧合流到 combined decision board，表示两条轨道最终进入统一决策与证据输出。这个结构本身已经构成了 Chapter 3 的主命题：**不同风险语义可以并行，但不能混淆；不同轨道可以独立，但不能割裂；最终目标不是双报告，而是统一治理。** [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

因此，本章必须明确系统**不应如何理解双轨评测**。它不应被误解为“全球评测 + 中国区本地补丁”，因为当前设计已经明确 TC260 轨道具有独立 test suite、aggregation logic、report template 以及规模化中文用例与细分风险类别；它也不应被误解为“两个孤立平台”，因为系统明确采用 shared backbone、统一 execution engine、结果存储与回溯能力，并在 combined decision board 合流；它更不应被误解为“为了兼顾业务而做的工程妥协”，因为当前设计已经把其研究升格方向明确表述为 Geo-aligned Safety Semantics 与 Compliance-to-Evaluation Compilation。也就是说，双轨评测不是实现细节，而是整个控制平面在评测层面对复杂风险现实的正式建模方式。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

综上所述，本章的核心结论是：**Dual-Track Evaluation Backbone 是 Unified AI Security Control Plane 的评测主干，它以 shared evaluation backbone 为基础，在同一执行与治理底座上同时承载 Global Safety & Robustness Track 与 China Regulatory Compliance Track（TC260），并通过不同 test suite、aggregation logic 与 report/output semantics，形成面向统一决策与证据输出的双来源风险测量体系。** 当前设计已经明确给出了这一结构的系统组成、功能分工与理论升格方向，包括双轨结构、共享执行引擎、Run / Target / Test Profile、API / CI-CD、结果存储与回溯、combined decision board、3572 条中文用例、31 个细分风险类别，以及 Geo-aligned Safety Semantics / Compliance-to-Evaluation Compilation 等研究命题。由此，Chapter 3 并不是对评测平台的技术描述，而是在蓝皮书中正式确立：企业级 AI 安全的评测基础设施，必须能够在统一控制平面内承载多语义、多地域、多治理目标的并行评测与统一治理。 [[统一 AI 安全控制平面 | PowerPoint]
