Executive Summary

本蓝皮书提出的，并不是一个由若干 detector、一个孤立 evaluation 平台、若干中文能力和若干合规功能拼接而成的工程集合，而是一套统一的 **AI Security Control Plane**：它将 **风险测量、策略判断、运行时执行、证据治理、发布放行以及中国与多语 intelligence** 组织为同一个系统边界下的连续能力体系。当前设计已经明确指出，DTL 不应被理解为若干 detector 或一个孤立 evaluation 平台，而应被理解为一个统一的 AI Security Control Plane；其中 **Evaluation** 负责风险测量与证据治理，**Runtime Kernel** 负责策略执行与可解释防护，**Chinese & Multilingual Safety Intelligence** 负责 geo-specific semantics，而 **xCloud** 负责 execution、orchestration 与 SLA。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这一定义意味着，企业级 AI 安全的核心问题不再是“是否拥有更多能力模块”，而是“是否建立了统一的安全判断与治理体系”。当前设计已经明确给出这一系统的中心结构：上方是 **Evaluation & Benchmark Governance**，中心是 **Security Control Plane**，左下是 **Chinese & Multilingual Safety Intelligence**，右侧是 **xCloud / Execution Environment**，右下是 **Evidence & Release Gating**。这一结构表明，安全系统的重点不是罗列功能，而是把 **benchmark、judgment、runtime control、evidence 与 release decision** 组织为同一控制平面中的不同层次。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

蓝皮书的第一个核心主张，是必须建立一套统一的 **Security Semantics、Benchmark Constitution 与 Acceptance Contract**。当前设计已经明确强调，所有安全判断都不能脱离 **benchmark 与 acceptance contract**；同时，系统需要突出的是 **统一语义层、统一判断层与统一证据层**，而不仅是单个 detector。材料还明确指出，Input Controls 已具备 **standardized detection categories and scoring**，这说明系统必须先把安全、隐私、内容风险、泄漏、攻击等对象组织为统一 taxonomy，再将其转化为评测、判断与放行逻辑。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

蓝皮书的第二个核心主张，是建立共享底座上的 **Dual-Track Evaluation Backbone**。当前设计已经明确指出，Evaluation 平台采用双轨结构：上轨为 **Global Safety & Robustness Track**，下轨为 **China Regulatory Compliance Track（TC260）**；两条轨道共享同一执行引擎、Run / Target / Test Profile、API / CI-CD、结果存储与回溯能力，但在 **test suite、aggregation logic 与 report/output template** 上保持差异，并最终汇入 **combined decision board**，进入统一决策与证据输出。材料同时明确写明，TC260 轨拥有 **3572 条全中文用例** 与 **31 个细分风险类别**，说明中国监管评测并不是全球安全评测的一层薄适配，而是一个具备独立语义空间与独立决策逻辑的正式轨道。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

蓝皮书的第三个核心主张，是把运行时安全正式定义为 **Runtime Security Kernel**，而不是单点 guard model。当前设计已经明确给出了运行时流水线：从 `detect_languages()` 开始，经由 **Normalization、Sanitization、Lite Detectors、Intelligent Gate、Heavy Detectors、Decision Layer**，最终输出 **Scoring、Aggregation、Allow / Block / Redact / Degrade** 等动作。材料还明确说明，当前系统采用的是 **rule-based、heuristics-based、proximity-based、regex-based、ML-based** 的混合架构，并要求在 xCloud 场景下提供 **explainable blocking reasons**。这表明，企业级 AI 安全的运行时层不应被理解为若干 detector 的堆叠，而应被理解为一个多阶段、可组合、可解释、可升级的安全执行内核。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

蓝皮书的第四个核心主张，是把 **Risk-Adaptive Control Selection** 建立为整套体系的方法论中枢。当前设计已经明确指出，Evaluation 与 Control 不是两套割裂系统，而是通过 **Measure → Profile → Select → Enforce** 形成闭环，并在右侧接入 **Evidence & Release Gating**，再反馈回 **Measure**。材料进一步明确列出了控制选择所依赖的关键因素：**base model capability、robustness against attacks、compliance / safety risk profile、model size & cost、latency & SLA、risk tolerance**，并直接将其理论升格方向表述为 **Risk-Adaptive Control Selection**，以及 **constrained optimization under SLA / cost / risk、selective classification / abstention、multi-stage decision under uncertainty、risk-aware policy selection**。这意味着，系统不应一刀切地采用最重控制，而应在风险、成本、时延与业务可用性约束下，选择最合适的控制级别。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

蓝皮书的第五个核心主张，是把 **Evidence Graph、Release Gating 与 Auditability** 提升为系统级治理核心。当前设计已经明确指出，系统真正需要突出的统一能力是 **从 benchmark → judgment → runtime API → evidence chain** 的连续能力；同时，判断结果必须进入 **Evidence & Release Gating**，形成 **release decision** 与 **audit artifact**，并反向反馈到 **Measure**，作为 regression 与持续改进的一部分。统一研发框架页又进一步把 **Evidence / CI-CD / Release Gating** 提升为 program 的一级能力单元。由此，证据体系不再是后置日志或审批材料，而是整个控制平面把判断、执行与发布治理制度化的中枢。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

蓝皮书的第六个核心主张，是将 **Chinese & Multilingual Safety Intelligence** 明确定义为全球 AI 安全体系中的核心 intelligence 对象。当前设计已经明确指出，中文与多语安全不是外围适配，而是 **intelligence 源头**；同时，TC260 轨作为独立评测轨道存在，运行时安全又以前置 **language gate** 与 **multi-language support** 为基础，而统一研发框架进一步要求中国团队参与 **Chinese-based detector development** 与 **Chinese-based evaluation and performance**。这说明，中国与多语能力不是部署末端的本地化支持，而是从安全语义、评测逻辑、运行时判断到放行治理全过程中的一级系统对象。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

在研发组织层面，蓝皮书最终提出的不是若干战术项目，而是一个统一的 **One DTL Security Program**。当前设计已经明确指出，系统突破、评测突破、中文/合规突破与理论突破，不应由割裂团队分别推进，而应在同一个统一 program 中协同完成；同时，整个 program 被组织为四个一级能力单元：**Security Science & Benchmark Governance、Runtime Security Kernel、Evidence / CI-CD / Release Gating、Chinese & Multilingual Safety Intelligence**。这一组织方式的意义，不在于资源重组本身，而在于为三条持续研究主线——**Risk-Adaptive Control Selection、Geo-aligned / Compliance-aligned Safety Semantics、Composable Security Kernel / Detector Cascade**——提供一个可持续演进的科研—工程母体。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

综合而言，本蓝皮书的总体主张可以被压缩为一句话：**企业级 AI 安全的目标，不是建设更多零散能力，而是构建一个统一的 AI Security Control Plane，使安全语义、双轨评测、运行时控制、风险驱动决策、证据治理以及中国与多语 intelligence 在同一系统边界下形成可持续强化的科研—工程闭环。** 当前设计已经明确给出了这一主张的系统基础、组织基础与理论基础，包括统一控制平面边界、双轨评测底座、运行时多阶段防护流水线、风险驱动闭环、证据与放行治理，以及统一 program 组织方式。由此，蓝皮书所提出的不是一个局部安全解决方案，而是一套面向下一代企业级 AI 部署的统一安全控制范式。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)
