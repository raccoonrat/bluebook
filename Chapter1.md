

Chapter 1. Unified AI Security Control Plane: Problem Statement and System Boundary

企业级 AI 安全的核心问题，不应被简化为“我们拥有多少 detector”“是否有一个 evaluation 平台”，或“是否已经具备若干局部合规能力”。这些表述都过于零散，无法刻画一个真实可部署、可治理、可持续强化的 AI 安全系统。根据当前设计，本项目真正需要回答的问题是：**我们到底在构建什么样的系统边界；谁负责安全判断，谁负责平台执行；评测、控制、证据、以及 xCloud 之间如何组成一个统一的 control plane。** 也正因如此，DTL 不应被理解为若干 detector 或一个孤立 evaluation 平台，而应被理解为一个统一的 **AI Security Control Plane**。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这一问题定义意味着，本蓝皮书从第一章开始，就必须将叙述对象从“功能组件列表”提升为“统一系统对象”。在当前设计中，这一系统对象具有明确的结构：上方是 **Evaluation & Benchmark Governance**，代表所有安全判断都不能脱离 benchmark 与 acceptance contract；中心是 **Security Control Plane**，承担真正的 routing、scoring 与 policy judgment；左下是 **Chinese & Multilingual Safety Intelligence**，强调中文与多语安全不是外围适配，而是 intelligence 源头；右侧是 **xCloud / Execution Environment**，承接 execution、orchestration 以及 SLA；右下则是 **Evidence & Release Gating**，负责把判断结果转化为可追踪的证据、放行决策与审计对象。这个布局说明，本系统的中心不是单一能力，而是一个把安全判断、执行承接与证据治理统一起来的控制平面。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

从这个意义上说，控制平面的提出，不是一个命名上的包装，而是对企业级 AI 安全问题的一次重新抽象。传统安全能力叙事通常把内容安全、攻击检测、模型防护、评测、审计、合规适配、多语言支持等问题拆散处理；而当前设计则明确指出，真正需要突出的关键技术，不是某个 detector 本身，而是**从 benchmark → judgment → runtime API → evidence chain 的统一系统化能力**。这意味着，安全并不止于检测，评测也不止于打分，运行时也不止于局部拦截；这些环节只有在同一个 control plane 中被组织起来，才能成为一个完整的企业级 AI 安全系统。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

在这一系统中，**Evaluation 的角色首先被重新定义。** 它不是一个独立于运行时之外的离线评分器，也不是简单地为模型输出若干 pass/fail 指标。当前设计已经明确指出，Evaluation 的职责是提供 **risk measurement and evidence governance**，并且所有安全判断都不能脱离 benchmark 与 acceptance contract。由此，Evaluation 成为整个系统的测量与治理入口：它定义可接受的风险边界，形成 benchmark constitution，支撑后续的判断逻辑，并最终服务于 release decision 与 audit。换言之，Evaluation 的价值，不在于输出一组分数，而在于为整个控制平面提供可以被引用、被追踪、被审计的判断依据。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

与此相对应，**Runtime Kernel 的角色也被重新定义。** 当前设计明确给出了运行时防护的工业化流水线：系统并不是“有一个 guard model”这么简单，而是包含 `detect_languages()`、Normalization、Sanitization、Lite Detectors、Intelligent Gate、Heavy Detectors 与 Decision Layer 的多阶段 pipeline；同时，detector inventory 也表明当前体系采用的是 **rule-based、heuristics-based、proximity-based、regex-based、ML-based** 的混合架构，而不是单一模型方案。最终，运行时层输出的不只是 allow / block 的二元结果，而是包含 scoring、aggregation、explainability 以及 allow / block / redact / degrade 等动作的综合判断。由此可见，Runtime Kernel 不是一个局部执行器，而是一个由多阶段检测、智能路由、升级决策与解释性动作构成的安全执行内核。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

然而，单独拥有 Evaluation 与 Runtime 仍然不足以构成真正的控制平面。当前设计之所以强调 **Evidence & Release Gating**，正是因为安全判断只有被转化为可追踪的证据与放行逻辑时，才真正具备治理意义。现有材料已经明确提出：判断结果最终必须进入统一证据层，成为 release gating 与 audit artifact 的组成部分；也就是说，系统关注的不是某次单点检测是否触发，而是整个 benchmark、judgment、runtime API 与 evidence chain 能否形成连续闭环。这样一来，安全系统就不再只是做出局部决定，而是能够为发布、审计、追责与持续改进提供制度化依据。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

这一闭环在“Risk Adaptation”部分得到了更进一步的表达。当前设计明确把整个流程概括为 **Measure → Profile → Select → Enforce**，并在右侧连接 **Evidence & Release Gating**，再将其反向反馈回 Measure，用于 regression 与持续改进。这里最重要的变化在于：Evaluation 不再只是离线测分工具，而被定义为 **decision signal source**；Control 也不再是固定配置，而是依据 **base model capability、robustness against attacks、compliance / safety risk profile、model size & cost、latency & SLA、risk tolerance** 来做控制级别选择；Runtime 则负责将这些决策真正执行为 allow、block、redact 或 degrade 等动作。由此，控制平面不仅是一个系统结构，更是一套风险驱动的持续强化机制。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

在这个统一结构中，**Chinese & Multilingual Safety Intelligence** 的位置具有特别重要的意义。当前设计明确强调，中文与多语安全不是外围适配，而是 intelligence 源头；与此同时，在双轨评测框架中，**China Regulatory Compliance Track（TC260）** 并非对全球安全评测的一层薄适配，而是与 **Global Safety & Robustness Track** 并行存在、共享底座但保持独立 test suite、aggregation logic 与 report template 的完整轨道。材料还明确指出，TC260 轨具有规模化中文用例与细分风险类别，这说明中国语义与多语能力并不是附属性能，而是会直接改变评测逻辑、判断逻辑与放行逻辑的核心建模对象。也正因如此，中国与多语能力必须位于控制平面的主系统内部，而不能被降格为区域支持层。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

上述结构进一步要求我们明确 **DTL 与 xCloud 的责任边界**。当前设计已经反复强调：**DTL 提供 security judgment / evidence，xCloud 提供 execution / orchestration / SLA。** 这一界定至关重要，因为它把“安全判断”和“平台执行”区分为两个不同但耦合的责任面。DTL 的职责，不是替代平台执行环境，而是为平台提供可解释的安全判断、评分体系、策略建议、阻断原因以及证据输出；xCloud 的职责，则是承接这些判断，在平台侧完成执行、编排、接口交付与 SLA 管理。这一边界既避免了系统设计的职责膨胀，也使控制平面真正成为“安全判断与证据治理的中心”，而不是一个泛化的基础设施替代方案。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

因此，本章必须明确指出：本系统**不是什么**。它不是 detector 的集合，因为现有设计已经强调真正的重点是统一语义层、统一判断层与统一证据层，而不是单个 detector；它不是孤立的 evaluation 平台，因为 Evaluation 的职责已经被界定为服务于 release decision、责任边界与审计，而不是停留在 run / target / result store 的层面；它也不是一个区域化 support 体系，因为 Chinese & Multilingual Safety Intelligence 在当前设计中被明确置于控制平面的核心，而非外围适配位置。只有把这些误解提前排除，本蓝皮书后续章节才能在同一个理论对象之下展开。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

从理论上看，这一章最重要的价值，在于把系统架构问题升格为统一的研究对象。当前设计已经明确提出，最适合的理论表达是 **AI Security Control Plane**：其价值不在于“多个模块怎么连”，而在于三类更深层的问题。第一，如何形式化**安全判断**与**执行编排**之间的责任边界；第二，如何把 **benchmark 证据、运行时策略与审计记录** 放入同一个可组合框架；第三，如何把 **geo-specific intelligence** 纳入统一控制平面。围绕这一统一对象，当前设计进一步点明了三条核心理论主线：其一是 **risk-adaptive control selection**，即将控制选择提升为在风险、成本、时延与可用性约束下的决策问题；其二是 **composable defense cascade**，即将运行时安全理解为异构检测、路由与动作执行构成的多阶段防御级联；其三是 **compliance-aligned safety semantics**，即把地域、法规与语言差异视为安全语义建模的一部分，而非外部附加条件。正是这三条主线，使本系统从“高水平工程组件拼装”进一步升格为“统一的安全控制理论框架”。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)

综上所述，本章的核心结论是：**Unified AI Security Control Plane** 不是一个由 detector、evaluation、合规适配与日志报表构成的工程集合，而是一个围绕安全判断、风险测量、运行时执行、证据治理与发布放行建立的统一系统。其系统边界已经在当前设计中被明确：Evaluation 负责 benchmark governance 与风险测量，Security Control Plane 负责 routing、scoring 与 policy judgment，Chinese & Multilingual Safety Intelligence 负责 geo-specific semantics，xCloud 负责 execution / orchestration / SLA，Evidence & Release Gating 则负责把判断结果转化为可追踪、可审计、可放行的结果。这一边界定义，不只是后续章节的结构起点，更是整本蓝皮书的第一性系统对象。后续关于双轨评测底座、运行时安全内核、风险自适应控制、证据图谱以及中国/多语 intelligence 的全部展开，都应被理解为这一统一控制平面的不同展开，而不是彼此平行的研发条线。 [[统一 AI 安全控制平面 | PowerPoint]](https://lenovo-my.sharepoint.com/personal/wangyh43_lenovo_com/_layouts/15/Doc.aspx?sourcedoc=%7BC99A9A8B-A736-4945-830E-C325133B6A4B%7D&file=%E7%BB%9F%E4%B8%80%20AI%20%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%B9%B3%E9%9D%A2.pptx&action=edit&mobileredirect=true)
